      SLA и эскалация

      Пороговые значения и флаги для SLA-first triage и серверной авто-эскалации критичных обращений.

      Используется для full autopilot: критичные assigned-кейсы можно перераспределять по SLA-routing правилам.
      Используется правилами с required_assignee_skills и required_assignee_queues для маршрутизации по навыкам/очередям.
      Поддерживаются фильтры: channel/business/location/category/request_prefix/client_status/rating/unread/minutes_left/sla_state + exclude_category/request_prefix. Для маршрутизации доступны assign_to, assign_to_pool и assign_to_pool_strategy (hash_by_ticket/round_robin/least_loaded).
      Нужно для e2e SLA-orchestration: webhook будет эскалировать critical тикеты даже при назначенном ответственном.
      Список через запятую, от 1 до 12 значений, диапазон 5..1440 минут.
      Пусто = SLA: все.
      Длительность действия «Отложить» в triage и bulk.
      Workspace runtime-параметры

      Закрывает backlog по production-parity: параметры надёжности workspace теперь настраиваются без правок кода.

      Используйте только после подтверждённого функционального parity workspace. При ошибках auto-fallback в модалку недоступен.
      Разрешённые блоки: messages, context, sla, permissions. Пусто = безопасный дефолт.
      Применяется к первичной загрузке и кнопке «Загрузить ещё».
      Используйте для rollout-проверки паритета: при ошибках workspace остаётся в текущем экране и не выполняет auto-fallback.
      После N ошибок workspace временно отключается для текущей сессии.
      Формат: {"operator_identity":"test|control"}. Невалидные значения игнорируются.
      Guardrails и decisioning

      Настраиваемые пороги для rollout telemetry и минимальные выборки для cohort/segment decision.

      Default переменные макросов

      Резервные значения для переменных {{name}} в macro dry-run и при реальном применении.

      Формат: {"ключ":"значение"}. Ключи нормализуются в lowercase. Пустые значения игнорируются.
      Каталог macro-переменных

      Единый каталог для внешних систем: ключ, label и default-подстановка для предпросмотра.

      Формат элемента: {"key":"variable_name","label":"Название","default_value":"опционально"}.
      Опционально: API каталога переменных. Поддерживается ответ в формате [...] или {"variables":[...]}.
      0 = без кэша. При недоступности внешнего каталога используется stale cache (если есть).
      Опционально: заголовок авторизации для защищённого external catalog API.
      Внешние профили клиента (workspace)

      Позволяет подключить CRM/contract-системы и дополнительные enterprise-ссылки без code-deploy.

      Поддерживаемые placeholder: {ticket_id}, {client_id}, {client_name}, {phone}, {channel_id}, {channel_name}, {business}, {location}.
      Опционально: JSON API для enrichment профиля. Поддерживаемые форматы: {...} или {"profile": {...}}.
      Опционально: заголовок авторизации для внешнего API enrichment (например, Authorization: Bearer ...).
      Если пусто — поведение workspace не меняется. Поддерживаемые ключи: crm, contract, external, local.
      0 = freshness-check выключен.
      Задаёт порядок поиска timestamp-полей. Невалидные значения безопасно попадают в invalid_utc.
      Опционально переопределяет общий TTL по конкретным источникам. Значения — часы в UTC, диапазон 0..8760.
      Mandatory customer profile (workspace)

      Позволяет формализовать минимальный профиль клиента: общий baseline и дополнительные обязательные поля для конкретных сегментов кейсов. Если настройки пустые — поведение workspace не меняется.

      Применяются ко всем кейсам. Ключи нормализуются в lowercase, дубликаты игнорируются.
      Ключ — сегмент из workspace, значение — список обязательных атрибутов для этого сценария.
      Enterprise-атрибуты клиента (workspace)

      Управление enrichment-полями в правой панели: можно скрыть шумные атрибуты, задать человеко-понятные названия и порядок отображения.

      Формат: ["attribute_key"]. Ключи нормализуются в lowercase, пустые значения игнорируются.
      Формат: {"attribute_key":"Название в UI"}. Помогает привести enterprise-ключи к бизнес-терминам.
      Формат: ["attribute_key"]. Указанные ключи будут показываться первыми в заданном порядке.
      Ключи с этими префиксами будут скрываться при включенном переключателе «Скрывать тех. ключи».
      Контекст и risk-сегменты workspace

      Тонкая настройка объёма правой панели и порогов авто-сегментации клиента без code-deploy.

      Сегмент reactivation_risk активируется, если total_dialogs ≥ порога и resolved_30d ≤ max значения. Сегмент open_backlog_pressure — если open_dialogs ≥ порога и доля open-диалогов превышает заданный процент.
      Кросс-продуктовые KPI-дашборды (BI bridge)

      Ссылки на внешние omni-channel/финансовые доски для закрытия пробела roadmap без отдельного релиза.

      Если включено, решение rollout будет на hold, пока оба чекпоинта внешних KPI не подтверждены.
      Контакт нужен для быстрого эскалационного пинга по внешнему BI blocker без поиска владельца вручную.
      Если включено, plain-текст без канала связи не снимает hold для внешнего dependency-ticket.
      Если ссылка на внешний тикет давно не обновлялась, rollout остаётся в hold до обновления статуса зависимости.
      Проверка UTC timestamps…
      UTC. Пустое значение оставляет freshness-gate выключенным, невалидное — будет безопасно проигнорировано и удержит hold только при включённом gate.
      UTC timestamp review-checkpoint.
      Если включено, rollout decision будет hold, пока не указано свежее время обновления внешних KPI.
      UTC timestamp последнего обновления внешней витрины.
      Позволяет не масштабировать rollout, пока BI-ссылки не заданы и операторы не имеют доступа к внешним витринам.
      Если статус health не обновлялся дольше TTL, rollout остаётся в hold даже при состоянии healthy.
      UTC timestamp обновления health-сигнала.
      Помогает формально учитывать внешний платформенный backlog по data-mart в rollout decisioning.
      Рекомендуется указывать при статусе blocked; при включённом blocker-gate ссылка обязательна и должна быть корректной (http/https).
      При включении rollout удерживается в hold, если статус программы не обновлялся дольше TTL.
      UTC timestamp обновления программного статуса.
      Если статус программы не `ready`, решение rollout перейдёт в hold после дедлайна + grace-период.
      UTC deadline готовности data-mart.
      Фиксирует минимальный KPI-контракт (mandatory/optional/available) и удерживает rollout в hold при отсутствии обязательных метрик.
      Позволяет удерживать rollout в hold, если покрытие optional-полей ниже заданного порога (без смешивания с mandatory-блокерами).
      Заполните mandatory/optional/available поля, чтобы увидеть покрытие data contract и влияние на rollout gate.
