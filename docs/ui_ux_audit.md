# UI/UX аудит панели Iguana

## Цель документа
Этот документ приведён к рабочему формату: фиксирует, **что уже сделано**, **что реально осталось в backlog**, и **какой следующий шаг даёт максимальный эффект**. Из файла удалены дубли, устаревшие промежуточные статусы и избыточная детализация уровня changelog.

## Что уже реализовано

### 1. Триаж очереди диалогов
- Добавлены saved views для основных сценариев работы: **Все**, **Активные**, **Новые**, **Без ответственного**, **Просроченные**, **SLA критичные**.
- Добавлены quick actions в списке диалогов: **назначить на себя**, **отложить**, **закрыть**.
- Реализованы массовые действия по выбранным диалогам.
- Добавлены горячие клавиши для triage, переключения views и массовых операций.

### 2. SLA и приоритизация
- В списке диалогов внедрён SLA-бейдж.
- Поддержан конфигурируемый SLA через `dialog_config` с безопасными fallback-значениями.
- Добавлены SLA-first фильтрация и сортировка на уровне UI.
- Введена базовая серверная поддержка SLA-orchestration и аудит действий.

### 3. Workspace и операторский поток
- Для `workspace_v1` реализован контракт загрузки карточки диалога.
- Добавлен fallback в legacy modal-flow при ошибках контракта, рендера или загрузки.
- В workspace есть базовый shell: лента сообщений, composer, правая контекстная колонка, telemetry и rollback через feature flag.

### 4. Макросы и governance
- Поддерживаются шаблоны ответов с переменными и preview.
- Реализованы базовые multi-step workflow-макросы.
- Добавлены аудит применения, версионность и ограничения на публикацию/approve.

### 5. Эксперименты и наблюдаемость
- Включены feature flags и A/B-сегментация операторов.
- Есть базовый telemetry-контур и guardrails для rollout workspace.
- Добавлены сводные метрики и уведомления по отклонениям.

## Текущее состояние
С точки зрения продукта основная база triage уже собрана: список диалогов стал быстрее для оператора, появился SLA-контур, есть foundation для workspace и управляемого rollout.

Главная проблема документа в предыдущем виде: он смешивал **аудит**, **roadmap**, **историю внедрения**, **список микро-изменений** и **спецификацию** в одном файле. Из-за этого терялась управляемость: важные незавершённые задачи тонули в длинном списке уже закрытых пунктов.

## Невыполненные задачи
Ниже только те направления, которые по состоянию документа остаются незавершёнными или закрыты частично.

### P1 — обязательно довести до production-grade

#### 1. Workspace как основной рабочий режим
**Статус:** частично выполнено.

Что уже есть:
- `workspace_v1` доступен;
- есть fallback в legacy modal-flow;
- есть базовый shell, composer и правая колонка.

Что не завершено:
- нет подтверждённого полного **functional parity** с legacy-сценарием;
- rollout всё ещё требует аккуратного контроля по cohort/feature flag;
- нужно окончательно убрать зависимость ключевого operator-flow от `dialogDetailsModal`.

**Что нужно сделать:**
- закрыть parity-gap между workspace и legacy;
- определить критерии полного перевода операторов на workspace;
- оставить modal только как rollback-механизм, а не как второй основной сценарий.

#### 2. Контекстная панель клиента
**Статус:** частично выполнено.

Что уже есть:
- блоки клиента, истории, связанных событий и SLA;
- независимые loading/error-state;
- базовый enrichment профиля.

Что не завершено:
- не подключены внешние источники уровня CRM/contract profile;
- не хватает расширенных enterprise-атрибутов и унифицированного customer context;
- контекст всё ещё ограничен локально доступными данными.

**Что нужно сделать:**
- определить минимальный обязательный customer profile для оператора;
- подключить внешние источники данных без перегрузки UI;
- стандартизировать приоритет показа контекстных блоков.

### P2 — важные, но не блокирующие для текущего rollout

#### 3. SLA-first orchestration на уровне сервера
**Статус:** частично выполнено.

Что уже есть:
- SLA-first фильтрация и сортировка;
- критичные views;
- часть авто-эскалации, auto-pin и routing-логики;
- аудит и базовый automation bridge.

Что не завершено:
- нет полного policy-слоя распределения по навыкам, очередям и инцидентным сценариям;
- не завершена зрелая авто-маршрутизация и авто-эскалация production-уровня;
- остаётся риск усложнения правил без единого центра управления.

**Что нужно сделать:**
- собрать единый SLA routing policy layer;
- описать и ограничить правила авто-назначения;
- разделить UI-приоритизацию и server-side orchestration по зонам ответственности.

#### 4. Макросы и централизованная библиотека переменных
**Статус:** частично выполнено.

Что уже есть:
- шаблоны, preview, workflow-шаги, аудит, governance.

Что не завершено:
- нет единого каталога переменных из внешних систем;
- не решён вопрос default values и унификации источников данных;
- не сформирована полноценная операционная модель сопровождения большой библиотеки макросов.

**Что нужно сделать:**
- выделить registry переменных;
- стандартизировать fallback/default policy;
- определить lifecycle макросов: draft → review → publish → archive.

#### 5. A/B-эксперименты по продуктовым KPI
**Статус:** частично выполнено.

Что уже есть:
- telemetry;
- cohort segmentation;
- guardrails и часть decisioning-механики.

Что не завершено:
- не замкнут контур на ключевые продуктовые KPI: **FRT**, **TTR**, **SLA breach %**, **CSAT**;
- нет полноценной decision-модели, основанной на продуктовых результатах, а не только на технических guardrails;
- не описан единый процесс принятия решения: rollout / hold / rollback.

**Что нужно сделать:**
- зафиксировать primary/secondary KPI для UX-изменений;
- связать telemetry workspace с продуктовыми метриками;
- ввести формальный rollout scorecard.

## Приоритетный план действий

### Следующий шаг с максимальным ROI
**Завершить перевод операторского потока на workspace без потери стабильности.**

Почему это приоритет №1:
- пока workspace не доведён до полного parity, команда вынуждена поддерживать два сценария работы;
- двойной контур увеличивает стоимость сопровождения, тестирования и обучения операторов;
- все последующие улучшения контекста, макросов и SLA-orchestration эффективнее развивать уже вокруг одного основного UX-потока.

### Рекомендуемая последовательность
1. **Закрыть parity workspace vs legacy**.
2. **Доукомплектовать контекстную панель клиента**.
3. **Собрать единый policy-layer для SLA orchestration**.
4. **Достроить registry переменных и governance макросов**.
5. **Связать rollout-решения с продуктовыми KPI, а не только с техническими guardrails**.

## Риски
- **Двойной UX-контур** ведёт к росту сложности поддержки и расхождению поведения между legacy и workspace.
- **Разрастание SLA-правил** без единого policy-layer усложнит сопровождение и аудит.
- **Рост библиотеки макросов** без строгого governance приведёт к дублированию, ошибкам и потере качества ответов.
- **Telemetry без продуктового decision-loop** даёт наблюдаемость, но не даёт управляемого принятия решений.

## Definition of Done для следующего этапа
Следующий этап можно считать завершённым, если одновременно выполняются условия:
- workspace покрывает ключевые ежедневные сценарии операторов без необходимости возврата в legacy modal-flow;
- modal остаётся только как rollback-опция;
- customer context показывает минимум данных, достаточный для принятия решения без переключения в сторонние экраны;
- SLA-критичные кейсы управляются не только визуально в UI, но и через формализованные server-side policy;
- rollout новых UX-изменений оценивается по FRT/TTR/SLA breach/CSAT, а не только по техническим метрикам.

## Итог
По факту проект уже вышел из стадии «просто UI-аудита»: базовые UX-улучшения внедрены, а главный незавершённый блок — это **доведение workspace, SLA orchestration и product-grade decisioning до целостной системы**.

Если держать фокус как senior-команда, то цель следующего цикла не в добавлении новых точечных фич, а в **сокращении архитектурного и операционного долга**: убрать двойные сценарии, собрать единые правила оркестрации и привязать rollout к реальным продуктовым результатам.
