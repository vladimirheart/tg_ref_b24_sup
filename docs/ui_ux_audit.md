# UI/UX аудит панели Iguana

## Цель документа
Этот документ переведён из формата «исторического списка хотелок» в управляемый senior-level статус-аудит: что уже доведено до рабочего состояния в кодовой базе, что закрыто частично и какие шаги реально уменьшают архитектурный и операционный риск в следующем цикле.

## Обновлённая картина по факту реализации
По состоянию текущего репозитория проект заметно продвинулся дальше, чем это было отражено в предыдущей версии аудита:
- triage-контур, SLA-индикация и quick actions уже являются базовым слоем продукта;
- `workspace_v1` больше не выглядит как «экспериментальный shell»: в конфигурации и API уже заложен режим **workspace primary** с возможностью жёстко отключать legacy fallback;
- customer context вышел за пределы локальных данных: есть поддержка внешнего profile enrichment, приоритета источников и деградации через cache/fallback;
- SLA orchestration вырос из простого UI-first подхода в серверную policy-модель с auto-assign rules, skill/queue matching и incident webhook-эскалацией;
- rollout-решения уже частично опираются не только на guardrails, но и на outcome KPI / scorecard / external KPI gate.

То есть главный вопрос уже не в том, «нужно ли это проектировать», а в том, **как довести существующие механизмы до предсказуемого production-режима без роста сложности сопровождения**.

## Что уже реализовано

### 1. Триаж очереди диалогов — реализовано
- Добавлены saved views для основных сценариев: **Все**, **Активные**, **Новые**, **Без ответственного**, **Просроченные**, **SLA критичные**.
- Добавлены quick actions в списке диалогов: **назначить на себя**, **отложить**, **закрыть**.
- Реализованы массовые действия по выбранным диалогам.
- Добавлены горячие клавиши для triage, переключения views и массовых операций.

### 2. SLA и приоритизация — реализовано базово, orchestration существенно усилен
- В списке диалогов есть SLA-бейдж.
- Поддержан конфигурируемый SLA через `dialog_config` с fallback-значениями.
- Добавлены SLA-first фильтрация и сортировка на уровне UI.
- Реализована серверная SLA-эскалация через webhook.
- Введены auto-assign rules с JSON-правилами, strategy для pool routing (`hash_by_ticket` / `round_robin` / `least_loaded`), а также матрицы навыков и очередей операторов.

### 3. Workspace и операторский поток — почти доведено до основного режима
- Для `workspace_v1` реализован контракт загрузки карточки диалога.
- Есть fallback в legacy modal-flow при ошибках контракта, рендера или загрузки.
- В workspace есть shell: лента сообщений, composer, правая контекстная колонка, telemetry и rollback через feature flag.
- В конфигурации уже предусмотрен режим, где workspace становится primary-flow, а legacy modal можно принудительно деактивировать как штатный UX-сценарий.
- Для rollback есть отдельный runbook с контрольными telemetry-метриками и fast rollback-процедурой.

### 4. Контекстная панель клиента — продвинута дальше, чем считалось ранее
- В правой колонке есть блоки клиента, истории, связанных событий и SLA.
- Поддержаны независимые loading/error-state.
- Есть базовый enrichment профиля.
- Добавлен внешний profile enrichment по URL-шаблону с auth-header, timeout, cache TTL и безопасной деградацией на stale cache.
- Добавлен приоритет источников customer context (`local`, `crm`, `contract`, `external`) и настройки для stale-control/extra attributes.

### 5. Макросы и governance — уже не только шаблоны
- Поддерживаются шаблоны ответов с переменными и preview.
- Реализованы multi-step workflow-макросы.
- Добавлены аудит применения, версионность и ограничения на публикацию/approve.
- Есть registry переменных: встроенные переменные, `dialog_config` catalog, default values и внешний catalog через URL.
- Макросы уже встроены в telemetry-контур как отдельный тип событий.

### 6. Эксперименты, telemetry и rollout decisioning — foundation собран
- Включены feature flags и A/B-сегментация операторов.
- Есть telemetry-контур по workspace-операциям и макросам.
- Есть guardrails по `workspace_open_ms`, `workspace_render_error`, `workspace_fallback_to_legacy`, abandon/slow-open.
- Есть `/api/dialogs/workspace-telemetry/summary`, breakdown по cohort/shift/team, а также gap breakdown по parity/context-source.
- Появился rollout scorecard.
- В конфигурации предусмотрен external KPI gate для проверки готовности внешних data/KPI-зависимостей перед rollout-решениями.

## Что изменилось по статусам
Ниже — пересмотр прежних «частично выполнено» на основании текущей реализации.

### Статус, который можно считать закрытым или близким к закрытию

#### 1. Workspace как основной рабочий режим
**Новый статус:** **в основном реализовано, остался операционный hardening**.

Что подтверждается сейчас:
- есть режим `workspace_primary` и cohort-rollout meta;
- есть флаги `workspace_force_workspace`, `workspace_decommission_legacy_modal`, `workspace_disable_legacy_fallback`;
- API уже различает, доступен ли legacy fallback и является ли он rollback-механизмом или полноценным вторым сценарием.

Что ещё не стоит считать полностью закрытым:
- functional parity должна подтверждаться не только кодом, но и эксплуатационной статистикой по parity-gap событиям;
- отключение legacy как UX-сценария требует контролируемого завершения миграции операторов, а не только выставления default-флагов;
- нужен явный список legacy-only сценариев, если такие ещё остались.

**Вывод:** задача больше не про «построить workspace», а про **доказать и закрепить его как единственный основной поток**.

#### 2. Контекстная панель клиента
**Новый статус:** **частично закрыто на уровне платформы, не закрыто на уровне данных и продуктовой полноты**.

Что подтверждается сейчас:
- внешние источники уже можно подключать;
- есть кэширование, auth и деградация при сбоях;
- есть приоритизация источников customer context.

Что остаётся:
- не доказан production-ready контракт для enterprise-данных из CRM/contract систем;
- не формализован минимально обязательный customer profile по ролям операторов и типам кейсов;
- остаётся риск визуальной перегрузки при расширении набора атрибутов.

**Вывод:** архитектурный каркас уже есть; следующий этап — **качество данных, обязательный профиль и правила показа**, а не изобретение новой панели.

#### 3. Макросы и централизованная библиотека переменных
**Новый статус:** **большая часть фундамента реализована**.

Что подтверждается сейчас:
- есть catalog переменных из settings и external source;
- есть default values;
- есть governance через approve/publish constraints и telemetry/audit.

Что остаётся:
- lifecycle макросов ещё нужно закрепить операционно: SLA на review, правила архивирования, owner per domain;
- нужен контроль деградации внешнего catalog и совместимости переменных между командами/каналами;
- стоит предотвратить будущий entropy в библиотеке через ownership и usage-based cleanup.

**Вывод:** здесь уже не backlog на разработку «registry», а backlog на **операционную модель библиотеки**.

#### 4. A/B-эксперименты по продуктовым KPI
**Новый статус:** **частично закрыто, decisioning стал значительно зрелее**.

Что подтверждается сейчас:
- telemetry умеет принимать KPI-driven события;
- есть primary/secondary KPI и required outcome KPI;
- есть rollout scorecard;
- есть external KPI gate для проверки внешней аналитической готовности.

Что остаётся:
- нужен единый регламент принятия решений rollout / hold / rollback, чтобы продукт, аналитика и инженерия интерпретировали scorecard одинаково;
- надо разделить «данные готовы» и «изменение действительно полезно для операторов/клиентов»;
- CSAT и часть внешних outcome-метрик всё ещё зависят от качества внешнего тракта и дисциплины заполнения.

**Вывод:** техническая база для KPI-driven rollout уже существует; следующий шаг — **сделать её управленческим стандартом**, а не просто набором сигналов.

## Что остаётся реально незавершённым

### P1 — production hardening и сокращение операционного долга

#### 1. Закрыть workspace parity не декларативно, а измеримо
**Проблема:** в коде уже есть telemetry для `workspace_parity_gap_events`, но это значит, что parity ещё должна подтверждаться цифрами, а не предполагаться.

**Что делать дальше:**
- зафиксировать список критичных сценариев parity: attachments, reply/edit, reopen, inline navigation, macros, SLA-actions, permissions edge-cases;
- завести weekly-review по parity-gap breakdown;
- выставить exit criteria: после N дней/недель без критичных parity-gap legacy modal перестаёт считаться рабочим UX-контуром.

#### 2. Довести customer context до product-grade стандарта
**Проблема:** инфраструктура источников уже есть, но нет достаточно строгого договора, какие данные обязательны для оператора в каждом сценарии.

**Что делать дальше:**
- определить minimal viable customer profile по сегментам кейсов;
- формализовать source-of-truth и freshness policy по каждому атрибуту;
- ограничить объём secondary-атрибутов и ввести UX-правила collapse/prioritization для предотвращения шумового интерфейса.

#### 3. Зафиксировать rollout governance для workspace
**Проблема:** флаги, telemetry summary, scorecard и external gate уже есть, но без формального decision-loop команда всё ещё может принимать rollout-решения вручную и неравномерно.

**Что делать дальше:**
- определить обязательный rollout packet: scorecard, parity-gap snapshot, external KPI gate status, incident history, owner sign-off;
- явно прописать пороги для `go / hold / rollback`;
- связать rollback runbook с decision model, чтобы эксплуатация и продукт работали по одной процедуре.

### P2 — развитие без роста хаоса

#### 4. Превратить SLA routing policy в управляемую систему, а не просто мощный JSON
**Проблема:** routing rules, skill/queue matrix и auto-assign уже есть, но при росте числа правил быстро появится конфигурационный долг.

**Что делать дальше:**
- описать policy layering: global rules → domain rules → emergency overrides;
- ограничить число конкурирующих условий и задать правило приоритета/конфликта;
- добавить регулярный аудит «неиспользуемых», «конфликтующих» и «слишком широких» routing rules;
- предусмотреть dry-run/preview режим для новых SLA rules перед включением в autopilot.

#### 5. Перевести macro governance в ownership-модель
**Проблема:** техническая библиотека переменных и шаблонов уже есть, но без владельцев доменов она неизбежно начнёт разрастаться и дублироваться.

**Что делать дальше:**
- назначить owner для каждой группы макросов и variable namespace;
- добавить ревизию неиспользуемых макросов и deprecation-policy;
- контролировать breaking changes во внешнем catalog переменных.

#### 6. Замкнуть telemetry на постоянный improvement loop
**Проблема:** метрик уже много, но без регулярного ритуала часть сигналов останется «наблюдаемостью без действия».

**Что делать дальше:**
- закрепить weekly ops/product review по telemetry summary;
- разделить сигналы на: immediate incident, rollout blocker, backlog candidate;
- для каждого alert/gap задать owner и SLA реакции.

## Приоритетный план действий

### Следующий шаг с максимальным ROI
**Не добавлять новые UX-фичи, а завершить hardening текущего workspace-контура как единственного primary-flow.**

Почему это приоритет №1:
- технически foundation уже собран;
- главный remaining cost теперь создаёт не отсутствие фич, а coexistence сложной конфигурации, parity-gaps и ручных rollout-решений;
- пока primary-flow не закреплён метриками и governance, любая новая фича будет добавлять долг сразу в несколько слоёв: UI, telemetry, routing, support process.

### Рекомендуемая последовательность
1. **Закрыть измеримый parity-gap workspace vs legacy и формально завершить деактивацию legacy modal как основного UX-сценария.**
2. **Определить mandatory customer profile и правила приоритета источников/атрибутов.**
3. **Ввести rollout governance packet: scorecard + external KPI gate + parity snapshot + owner sign-off.**
4. **Нормализовать SLA policy layer и добавить практику регулярного policy-аудита.**
5. **Ввести ownership и cleanup-процесс для macro library / variable catalog.**

## Основные риски
- **Скрытый dual-run риск:** даже при primary workspace команда может фактически продолжать зависеть от legacy, если parity проверяется эпизодически.
- **Config debt в SLA orchestration:** чем больше JSON-правил без строгой модели владения и ревизии, тем выше риск непредсказуемого routing behavior.
- **Data debt в customer context:** наличие интеграции с внешним профилем не гарантирует качество, свежесть и полезность контекста.
- **Metric theater:** scorecard и guardrails не улучшают систему сами по себе, если по ним нет формального цикла решений.
- **Library entropy:** макросы и переменные без owner-модели быстро становятся трудноуправляемыми.

## Definition of Done для следующего этапа
Следующий этап можно считать завершённым, если одновременно выполняются условия:
- workspace подтверждён как основной ежедневный поток операторов не только флагами, но и telemetry без критичных parity-gap;
- legacy modal больше не используется как штатный сценарий работы и остаётся только rollback-опцией;
- customer context показывает обязательный набор данных для основных кейсов с понятной source/freshness policy;
- SLA-routing правила проходят регулярную ревизию и не создают неконтролируемого конфигурационного долга;
- rollout новых UX-изменений принимает решение по формальному scorecard/process, а не по интуиции команды;
- macro library имеет owner-модель, review-процесс и cleanup/deprecation-практику.

## Итог
Проект уже вышел из стадии «UI-аудита с набором идей». Базовые UX-улучшения, workspace foundation, customer context plumbing, SLA policy и KPI-driven telemetry **уже реализованы в значительной степени**.

Следующий цикл должен быть сфокусирован не на расширении поверхности функций, а на **систематизации уже созданной мощности**:
- закрепить workspace как единственный primary UX;
- довести контекст и маршрутизацию до quality-controlled production standard;
- превратить telemetry, scorecard и governance из набора инструментов в регулярную управленческую практику.

Именно это даст максимальный эффект senior-команды: **меньше скрытого долга, меньше расхождения сценариев, больше предсказуемости продукта в эксплуатации**.
