# Автоматизация Bitrix24 + iiko

Документ описывает runtime-контур панели для сценария отключения корпоративных
скидок сотрудников.

## Что уже умеет контур

- запускаться из панели по кнопке;
- выполнять `dry-run` без боевых изменений;
- использовать личные доступы текущего пользователя панели для Bitrix24 и iiko;
- хранить несколько iiko URL-профилей на пользователя и переключать активный профиль;
- загружать организации iiko для выбранного URL;
- загружать активные категории гостей и кошельки/программы корпоративного питания для выбранного URL;
- сохранять выбор категорий и кошельков внутри активного URL-профиля;
- подключаться к Bitrix24 по webhook;
- находить группы и задачи;
- фильтровать только незакрытые задачи, в которых checklist-пункт еще не
  отмечен;
- извлекать телефон сотрудника из тела задачи;
- по боевому запуску искать гостя по телефону и удалять выбранные категории и
  выбранные программы корпоративного питания;
- вести историю запусков и результатов по каждой задаче в БД панели.

## Где находится UI

- страница `AI Ops`;
- блок `Bitrix24 + iiko automation`.

## Где теперь хранятся секреты

Секреты больше не привязаны к local-machine файлам.

Личные доступы Bitrix24/iiko сохраняются в панели отдельно для каждого
пользователя через `settings_parameters` с ключом
`employee_discount_automation_credentials.v1`.

В UI секретные поля не возвращаются обратно в браузер:

- `webhook_url`;
- `api_secret`.

Если оставить эти поля пустыми при сохранении, панель сохраняет уже существующее
секретное значение.

## Что хранится в общих versioned settings

В `config/shared/settings.json` сохраняются только общие non-secret настройки
автоматизации:

- `bitrix_group_id`;
- `task_title_markers`;
- `checklist_labels`;
- `phone_regex`;
- `dry_run_by_default`.

## Что хранится в личном конфиге пользователя панели

Личный конфиг пользователя может содержать:

- `bitrix24.portal_url`;
- `bitrix24.webhook_url`;
- `active_iiko_profile_url`;
- набор `iiko_profiles` по `base_url`;
- для каждого URL-профиля:
  - `base_url`;
  - `api_login`;
  - `api_secret`;
  - `organization_id`;
  - `selected_discount_category_ids`;
  - `selected_wallet_ids`.

## Рабочая механика в панели

- шаг 0: пользователь сохраняет URL и креды для нужного iiko-контура;
- шаг 1: по активному URL загружаются организации, категории и кошельки;
- шаг 2: пользователь отмечает нужные категории и кошельки и сохраняет профиль;
- шаг 3: панель анализирует задачи Bitrix24 и для найденных сотрудников удаляет
  выбранные категории и кошельки/программы.

## Ограничения текущего этапа

- под “кошельками” сейчас используется documented-механика программ
  корпоративного питания через `remove_from_nutrition_organization`;
- перед production-использованием нужен живой прогон на реальных учетных
  данных и подтверждение, что бизнес-смысл “кошелька” у вас действительно
  соответствует этой iikoCard-модели;
- checklist в Bitrix24 отмечается только после успеха по задаче;
- персональные секреты уже user-scoped, но пока хранятся в БД панели без
  отдельного слоя шифрования.

## Что нужно для завершения боевого режима

- ручная проверка на ваших реальных URL и организациях;
- подтверждение, что выбранные “кошельки” действительно надо отключать через
  программы корпоративного питания;
- при необходимости тонкая корректировка обработчика под фактические тексты
  ошибок вашего iiko-контура.
