# План Развития AI-Агента Диалогов

## 1. Цель
Сделать AI-агента, который:
- автоматически обрабатывает типовые клиентские запросы;
- использует внутренние источники (база знаний, задачи, история диалогов);
- эскалирует оператору, если уверенности недостаточно;
- помогает оператору подсказками ответа;
- не блокирует работу оператора: оператор может взять диалог в любой момент.

## 2. Что уже реализовано (база)
- Статус диалога `в автоматической обработке` на backend + frontend.
- Состояние AI-обработки в отдельной таблице `ticket_ai_agent_state`.
- Базовый движок релевантности по источникам:
  - `knowledge_articles`
  - `tasks`
  - `chat_history`
- Автоответ при достаточной уверенности.
- Эскалация в оповещения операторов при низкой уверенности/ошибке.
- API подсказок для оператора: `GET /api/dialogs/{ticketId}/ai-suggestions`.
- UI-блок подсказок в карточке диалога.
- При действиях оператора (`reply/media/take/resolve/reopen`) AI-режим сбрасывается.
- Оператор может перехватить/взять диалог независимо от текущей AI-обработки.

### Инкремент от 2026-04-02
- Добавлена таблица памяти решений `ai_agent_solution_memory` (миграция `V22`).
- Агент использует память решений как дополнительный источник подсказок и автоответов.
- Ответы операторов сохраняются и обновляют память (learning loop).
- При значимом расхождении ответа оператора с предложением AI:
  - фиксируется `review_required`,
  - оператору отправляется запрос на внесение правок в решение.
- Добавлен операторский контур ревизии в диалоге:
  - `GET /api/dialogs/{ticketId}/ai-review`
  - `POST /api/dialogs/{ticketId}/ai-review/approve`
  - `POST /api/dialogs/{ticketId}/ai-review/reject`
  - в UI карточки диалога появились кнопки «Принять правку / Оставить текущее».

## 3. Архитектура целевого решения
## 3.1 Контур обработки
1. Входящее сообщение клиента -> watcher события.
2. Классификация: типовой/нетиповой запрос + оценка уверенности.
3. Поиск кандидатов ответа в источниках.
4. Ранжирование кандидатов (score + объяснение источника).
5. Решение:
   - auto-reply (если score >= порога),
   - suggest-only (если средняя уверенность),
   - escalate (если низкая уверенность/ошибка/запрос человека).
6. Логирование решения и причин.

## 3.2 Роли режимов
- `AUTO_REPLY`: агент сам отвечает.
- `ASSIST_ONLY`: агент даёт подсказки оператору, не отправляет клиенту.
- `ESCALATE_ONLY`: только уведомление оператору.

Переключение режима должно быть в `dialog_config` без релиза.

- [ ] Явно маркировать автоответы как `ai_agent` (а не `operator/system`).

  - штраф за конфликтующие источники.
- [ ] Ответ по шаблону:
  - короткий итог;

## 5. Контракты и данные
## 5.1 Новые поля в `dialog_config` (предложение)
- `ai_agent_enabled: boolean`
- `ai_agent_mode: string`
- `ai_agent_auto_reply_threshold: number` (0..1)
- `ai_agent_suggest_threshold: number` (0..1)
- `ai_agent_max_auto_replies_per_dialog: number`
- `ai_agent_cooldown_seconds: number`
- `ai_agent_require_operator_for_sensitive_topics: boolean`

## 5.2 Аудит-события (предложение)
- `ai_agent_decision_made`
- `ai_agent_auto_reply_sent`
- `ai_agent_suggestion_shown`
- `ai_agent_suggestion_applied`
- `ai_agent_escalated`
- `ai_agent_disabled_for_dialog`

## 6. Критерии готовности (DoD)
- Функционально:
  - оператор всегда может взять диалог;
  - режимы агента переключаются из настроек;
  - автоответ не уходит при низкой уверенности.
- Надёжность:
  - нет бесконечных циклов автоответов;
  - все решения агента логируются.
- UX:
  - у оператора есть понятные подсказки и причины выбора;
  - есть кнопки ручного контроля агента.
- Метрики:
  - доступны базовые KPI по качеству работы агента.

## 7. План спринта (1-2 недели)
1. Настройки режима/порогов агента в UI + backend.
2. Лимиты и cooldown автоответов.
3. Маркировка автоответа отдельным sender `ai_agent`.
4. Аудит-события по решениям агента.
5. Мини-дашборд AI-метрик в аналитике.

## 8. Риски и меры
- Ложные автоответы:
  - conservative thresholds;
  - режим `assist_only` по умолчанию на старте.
- Перегрузка операторов эскалациями:
  - батчирование/дедуп оповещений;
  - приоритизация по SLA и критичности.
- Регресс в скорости:
  - кэширование retrieval;
  - ограничение источников и размеров выборки.

## 9. Предложение по rollout
1. Этап 0: `assist_only` для всех, сбор метрик.
2. Этап 1: `auto_reply` только на whitelisted категориях.
3. Этап 2: расширение покрытия при стабильном CSAT и низком correction rate.
