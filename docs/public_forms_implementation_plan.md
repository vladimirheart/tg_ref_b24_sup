# План реализации внешних форм (до production-ready)

## 1) Цели и границы (что считаем «реализовано до конца»)

**Цель:** довести текущий зачаток публичной формы до полнофункционального и безопасного канала входящих обращений, интегрированного в текущий процесс диалогов/тикетов.

**Definition of Done:**
- Форма конфигурируется на уровне канала в панели (поля, обязательность, порядок, типы).
- Публичная ссылка стабильна, может быть отозвана/переиздана.
- Отправка формы создает корректный диалог/тикет в той же модели, что и боты.
- Клиент может открыть публичную страницу диалога по токену и видеть историю.
- Добавлены защита от спама/абьюза, аудит, метрики, тесты и runbook.

## 2) Текущее состояние (что уже есть)

- Есть backend-скелет: загрузка конфигурации формы, создание сессии, сохранение сообщения в `chat_history`.
- Есть публичная HTML-страница и JS-клиент.
- Есть API для `config`, `createSession`, `getSession`.
- Есть demo-режим, но отсутствует полноценный flow администрирования, защита и эксплуатационные механики.

## 3) Архитектурные решения (зафиксировать до разработки)

1. **Источник истины конфигурации формы**
   - Оставить `questions_cfg` как backward-compatible storage.
   - Ввести валидируемую schema-обертку (версия, список полей, UX-опции, антиспам-политика).

2. **Модель сессии/диалога**
   - Сессия формы = только точка входа.
   - Единый источник сообщений/статусов = существующая доменная модель диалогов.

3. **Безопасность ссылок**
   - Использовать публичный `channel public id` для URL формы.
   - Токен сессии хранить как одноразовый ключ доступа к публичной истории (с ротацией/TTL по политике).

4. **Расширяемость**
   - Типы полей описывать декларативно (text, textarea, select, checkbox, phone, email, file), чтобы без переписывания JS добавлять новые типы.

## 4) Этапы реализации

### Этап A. Конфигурирование формы в панели (MVP admin)

- Добавить в UI каналов раздел «Внешняя форма»:
  - переключатель «включить/выключить»;
  - редактор полей (drag/drop порядок, required, placeholder, help-text, options);
  - предпросмотр.
- Backend:
  - DTO + серверная валидация схемы конфигурации;
  - endpoint сохранения/чтения конфигурации;
  - versioning схемы (например, `schemaVersion=1`).
- Acceptance:
  - невалидный конфиг не сохраняется;
  - при отключении формы публичный URL возвращает 404/410 (по выбранной политике).

### Этап B. Публичная ссылка и lifecycle

- Добавить генерацию/ротацию публичной ссылки (publicId/regenerate).
- Политика отключения:
  - soft-disable (временно недоступна);
  - hard-revoke (новый publicId, старый недействителен).
- UI: «Скопировать ссылку», «Регенерировать», предупреждения о разрыве старых ссылок.

### Этап C. Валидация и UX формы на клиенте

- Рендерить поля из конфигурации, а не статически.
- Единая карта ошибок (frontend + backend): обязательные поля, формат email/phone, длины.
- Улучшить UX:
  - состояние загрузки кнопки;
  - retry/идемпотентность submit;
  - отображение номера обращения и короткой инструкции «что дальше».
- Локализация текстов ошибок/подсказок.

### Этап D. Интеграция с тикетингом/операционкой

- При submit создавать не только `chat_history`, но и полную сущность обращения в ожидаемой операционной цепочке:
  - категория/приоритет по правилам;
  - привязка к каналу;
  - SLA timestamp;
  - уведомление ответственных.
- Настроить автотеги (например, `source=web_form`, `urgency=...`).

### Этап E. Антиспам и безопасность

- Rate limit по IP + fingerprint + channel.
- CAPTCHA/Turnstile (включаемо по каналу).
- Санитизация HTML/скриптов, ограничения размеров payload.
- Threat model минимум: spam flood, token guessing, replay, CSRF/CORS misuse.
- Логи безопасности и алерты по аномалиям.

### Этап F. Файлы/вложения (если нужны бизнесом)

- Загрузка вложений (белый список MIME, антивирусный hook, лимиты размера/кол-ва).
- Хранение через существующий `AttachmentService` и отображение в истории.
- Политика retention для приватных данных.

### Этап G. Наблюдаемость и аналитика

- Метрики:
  - `form_views`, `form_submits`, `submit_errors`, `captcha_failures`, `time_to_first_response`.
- Дашборд conversion funnel: просмотр → заполнение → отправка → первый ответ.
- Аудит изменений конфигурации формы (кто/когда/что поменял).

### Этап H. Тестирование и стабилизация

- Unit-тесты:
  - парсинг/валидация конфигурации;
  - сборка итогового сообщения;
  - правила заполнения clientName/contact.
- Integration-тесты:
  - публичные endpoints, happy-path + негативные кейсы;
  - rate-limit/captcha ветки.
- E2E smoke:
  - создание формы в панели → submit с публичной страницы → тикет виден оператору.

### Этап I. Документация и релиз

- Обновить runbook:
  - включение формы;
  - регенерация ссылки;
  - разбор инцидентов «не приходят формы».
- Обновить пользовательскую документацию (админ-панель + публичный flow).
- Постепенный rollout (feature flag):
  1. internal channels;
  2. selected pilot;
  3. full rollout.

## 5) Потенциальные проблемы и превентивные меры

1. **Ломаем обратную совместимость старых `questions_cfg`**
   - Мера: миграционный адаптер + schemaVersion + fallback parser.
2. **Спам/флуд после публикации URL**
   - Мера: rate-limit, captcha, динамический blocklist.
3. **Рост нагрузки на БД (много анонимных обращений)**
   - Мера: ограничения, буферизация, архивирование, отдельные индексы по source/channel/time.
4. **Утечка персональных данных в логах**
   - Мера: маскирование PII в application logs + политика retention.
5. **Сложность поддержки динамического конструктора полей**
   - Мера: минимальный поддерживаемый набор типов в v1, расширение только по telemetry-driven backlog.

## 7) Оценка реализации (порядок и сроки)

- **Спринт 1:** Этапы A+B+C (базовая эксплуатационная готовность).
- **Спринт 2:** Этапы D+E (боевой поток и безопасность).
- **Спринт 3:** Этапы G+H+I + (F при необходимости).

## 8) Критерии приемки v1 (чек-лист релиза)

- [ ] Канал можно полностью настроить из UI без правок JSON вручную.
- [ ] Публичная ссылка управляется и отзывается админом.
- [ ] Все обязательные поля валидируются на backend.
- [ ] После submit оператор видит обращение в штатном рабочем интерфейсе.
- [ ] Включены rate-limit и базовая защита от automated spam.
- [ ] Есть метрики и runbook на инциденты.
- [ ] Покрытие критичного пути тестами (unit + integration + smoke).
