# Задача [01-143]

## Короткое описание

Переименовать канонические SQLite-файлы под их архитектурный смысл и сохранить runtime-совместимость.

## Цель

Сделать так, чтобы physical names основных канонических SQLite-файлов соответствовали их реальной architectural role, при этом проект не ломал существующие legacy-инсталляции и старые env-ключи в один шаг.

## Контекст

- В `01-142` уже зафиксировано target-state решение, где `tickets.db` по смыслу является `panel-runtime`, `users.db` — `panel-identity`, а `bot_database.db` — `bot-runtime`.
- Сейчас код, env defaults, bot runtime contract, тесты и документация по-прежнему опираются на legacy-имена `tickets.db`, `users.db` и `bot_database.db`.
- Прямой rename без compatibility-слоя рискован: часть runtime и bot-процессов всё ещё ожидает старые env keys (`APP_DB_TICKETS`, `APP_DB_USERS`, `APP_DB_BOT`) и старые filenames как fallback.
- Transitional secondary-файлы вроде `clients.db`, `knowledge_base.db`, `objects.db`, `settings.db` не входят в scope этого шага: их судьба должна решаться отдельными migration-задачами.

## Что сделать

- Ввести новые canonical filenames для основных контуров:
  - `panel_runtime.db`
  - `panel_identity.db`
  - `bot_runtime.db`
- Перевести `spring-panel` и `java-bot` на новые logical env keys и defaults с alias/fallback на legacy-ключи:
  - `APP_DB_PANEL_RUNTIME`
  - `APP_DB_PANEL_IDENTITY`
  - `APP_DB_BOT_RUNTIME`
- Сохранить runtime/backward compatibility:
  - старые env keys продолжают работать;
  - если рядом лежат старые physical filenames, runtime умеет их подхватить;
  - panel при запуске ботов прокидывает совместимый набор env-переменных.
- Обновить активную документацию и user-facing diagnostics под новый naming, явно пометив legacy-алиасы.
- Переименовать репозиторные канонические SQLite-файлы, которые соответствуют новым основным контурам.

## Критерии готовности

- По умолчанию проект создаёт/ищет `panel_runtime.db`, `panel_identity.db` и `bot_runtime.db`, а не legacy main filenames.
- Старые env keys и старые physical filenames остаются поддержаны как compatibility fallback.
- Bot runtime contract и bot-core понимают новый naming.
- Основная документация и runtime diagnostics отражают новый canonical naming.
- Задача оформлена в `ai-context`, а итог изменений зафиксирован отдельным changelog-файлом.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog` и включить в него пользовательский промпт, который привёл к изменению.
