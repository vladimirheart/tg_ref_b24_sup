# Задача [01-097]

Провести рефакторинг CSS в `spring-panel`, уменьшить хаотичность больших файлов `style.css`, `app.css`, `sidebar.css` и, если сборочный контур позволяет сделать это безопасно, перевести поддержку стилей на `scss`-источники с генерацией тех же публичных CSS-артефактов.

## Цель

- разложить стили по более понятной структуре без смены путей подключения в шаблонах;
- сократить сложность поддержки больших CSS-файлов за счёт модульных `scss`-источников;
- сохранить визуальный контракт UI и совместимость со сборкой `spring-panel`.

## Контекст

- сейчас шаблоны `spring-panel/src/main/resources/templates/**` напрямую подключают `/css/style.css`, `/css/sidebar.css` и `/css/app.css`;
- `style.css` и `app.css` разрослись, содержат несколько слоёв исторических override-блоков и плохо подходят для точечной доработки;
- в `pom.xml` пока нет явной поддержки компиляции `scss`, поэтому перевод возможен только при безопасном встраивании в существующий Maven pipeline.

## Что сделать

- завести структуру `scss`-источников в `spring-panel/src/main/resources`;
- подключить компиляцию `scss -> css`, не ломая текущие пути `/css/*.css`;
- перенести и сгруппировать существующие стили по смысловым модулям: токены, base, layout, components, pages, overrides;
- по возможности сократить очевидные дубли и сделать override-слои прозрачнее;
- проверить, что `spring-panel\\mvnw.cmd -q -DskipTests compile` проходит после изменений.

## Критерии готовности

- в проекте появляются поддерживаемые `scss`-источники и понятная структура стилей;
- итоговые `style.css`, `sidebar.css`, `app.css` продолжают подключаться без изменений шаблонов;
- сборка `spring-panel` проходит без ручных дополнительных шагов;
- финальный ответ содержит `Резюме для коммита`.
