# Задача [01-150]

## Короткое описание

Провести аудит legacy compatibility-слоя настроек бота/диалогов и спланировать его выпиливание.

## Цель

Подготовить безопасный план перехода к модели «один домен настроек = один источник истины» для bot settings, auto-close и dialog templates, чтобы дальше можно было поэтапно удалить старые поля, fallback-и и compatibility bridge без регрессий в runtime, UI и channel-level overrides.

## Контекст

- После аудита по багам шаблонов оценок, автозакрытия и вопросов подтвердилось, что проект одновременно поддерживает старую и новую модели настроек и местами синхронизирует их вручную.
- Сейчас один и тот же смысл живёт сразу в нескольких полях:
  - `bot_settings.question_templates` + `active_template_id` и зеркальный `bot_settings.question_flow`;
  - `bot_settings.rating_templates` + `active_rating_template_id` и зеркальный `bot_settings.rating_system`;
  - `auto_close_config.templates` + `active_template_id` и legacy fallback `auto_close_hours`;
  - `bot_settings.unblock_request_cooldown_minutes` и верхнеуровневый `unblock_request_cooldown_minutes`;
  - channel-level overrides в отдельных колонках (`question_template_id`, `rating_template_id`, `auto_action_template_id`) и legacy duplicate-пути внутри `channels.questions_cfg`.
- Есть отдельный legacy слой операторских шаблонов в `dialog_config.question_templates`, который относится не к bot settings, а к workspace/legacy dialog modal, но UI сейчас легко смешивает эти сущности в голове пользователя.
- Frontend и backend всё ещё держат compatibility fallback-ветки:
  - `spring-panel/src/main/resources/static/js/bot-settings.js` уже не использует legacy `BOT_SETTINGS_INITIAL` / `BOT_PRESET_DEFINITIONS`, но всё ещё держит diagnostics и совместимость вокруг imported legacy payload;
  - `java-bot/bot-core/src/main/java/com/example/supportbot/settings/BotSettingsService.java` строит sanitized output сразу с `question_flow` и `rating_system`;
  - `spring-panel/src/main/resources/static/js/settings-save-runtime.js` и `spring-panel/src/main/java/com/example/panel/service/SettingsTopLevelUpdateService.java` уже пишут только канонический `auto_close_config`, но cleanup read-fallback для `auto_close_hours` ещё остаётся на backend/runtime;
  - `spring-panel/src/main/java/com/example/panel/service/SettingsDialogTemplateConfigService.java` продолжает писать `dialog_question_templates` в `dialog_config.question_templates`;
  - `java-bot/bot-core/src/main/java/com/example/supportbot/service/MaintenanceTasks.java` держит legacy fallback на `auto_close_hours`;
  - `spring-panel/src/main/java/com/example/panel/service/ChannelTransportService.java` и channel runtimes по-прежнему используют `questions_cfg` как compatibility-контейнер.

## Аудит legacy-слоя

### 1. Legacy/duplicate поля shared settings

- `bot_settings.question_flow`
  - Смысл: активный flow бота.
  - Канонический источник уже есть: `bot_settings.question_templates[active_template_id].question_flow`.
  - Риск: ручная синхронизация ломается при редактировании шаблонов и даёт UI/runtime разные представления.

- `bot_settings.rating_system`
  - Смысл: активная система оценок.
  - Канонический источник уже есть: `bot_settings.rating_templates[active_rating_template_id]`.
  - Риск: ответы для `value=5` и другие изменения могут храниться в двух местах и расходиться.

- `auto_close_hours`
  - Смысл: порог автозакрытия по умолчанию.
  - Канонический источник уже есть: `auto_close_config.templates[active_template_id].hours`.
  - Риск: UI сохраняет новый блок, а backend/ops читают старое значение.

- `unblock_request_cooldown_minutes` на корне `settings.json`
  - Похоже на исторический дубль `bot_settings.unblock_request_cooldown_minutes`.
  - Нужен отдельный owner/usage audit перед удалением.

### 2. Legacy channel compatibility

- `channels.questions_cfg`
  - Исторически хранит смешанный JSON с разными channel/runtime-настройками.
  - Сейчас там могут жить:
    - старые ссылки на template selection (`active_template_id`, `question_template_id`, `active_rating_template_id`, `rating_template_id`);
    - notification routing и alert queue compatibility payload.
  - При этом канонические channel override уже вынесены в отдельные колонки таблицы `channels`.

### 3. Legacy operator dialog templates

- `dialog_config.question_templates`
  - Это не bot settings, а шаблоны типовых вопросов для операторского dialogs/workspace.
  - Они всё ещё читаются в `dialogs.js` через `window.DIALOG_CONFIG.question_templates`.
  - Риск не в самом существовании поля, а в том, что оно семантически пересекается по названию с `bot_settings.question_templates` и создаёт ложное ощущение одного домена.

### 4. Frontend compatibility bridges

- `bot-settings.js`
  - legacy globals fallback уже убран, но остаётся переходная диагностика imported legacy payload;
  - следующий фронтовый cleanup после этого этапа переносится в channel compatibility и page runtime contract.

- `settings-page-config-runtime.js`
  - общий helper `resolveConfigValue(... legacyKey)` поощряет продолжение dual-contract загрузки.

- `settings-save-runtime.js`
  - write-side mirror `auto_close_hours` уже убран;
  - panel save-boundary больше не восстанавливает `auto_close_hours` из active template при сохранении `auto_close_config`;
  - remaining cleanup смещён в runtime/read fallback и fixtures.

### 5. Backend compatibility bridges

- `BotSettingsService`
  - sanitization до сих пор считает `question_flow` и `rating_system` частью итогового контракта, а не чисто derived compatibility output.

- `MaintenanceTasks`
  - держит fallback на `auto_close_hours` для backward compatibility, что нормально на переходе, но это нужно сделать явно временной стадией с deprecation-планом.

## Что сделать

- Зафиксировать целевую каноническую модель настроек и явно назначить owner каждому домену:
  - bot onboarding/questions;
  - bot rating;
  - auto-close;
  - operator dialog/workspace templates;
  - channel overrides.
- Для каждого legacy/duplicate поля составить таблицу:
  - где хранится;
  - кто пишет;
  - кто читает;
  - это canonical / derived / deprecated / migration-only;
  - можно ли удалить сразу или нужен compatibility window.
- Разделить legacy-хвосты на 3 очереди выпиливания:
  - `shared settings` mirrors (`question_flow`, `rating_system`, `auto_close_hours`, root cooldown duplicate);
  - `channel questions_cfg` compatibility keys;
  - `frontend/runtime bridges` и legacy globals/fallback-и.
- Подготовить поэтапный migration plan:
  1. Ввести явную документацию canonical schema и deprecation-статусов.
  2. Перестать использовать deprecated поля как source of truth.
  3. Перестать писать deprecated поля при новых сохранениях, оставив read-only fallback на переходный период.
  4. Добавить диагностику/логирование, показывающее, когда runtime ещё зависит от deprecated поля.
  5. После стабилизации удалить fallback reads и schema-level legacy keys.
- Отдельно описать границы между `dialog_config.question_templates` и `bot_settings.question_templates`:
  - что остаётся жить как отдельный домен;
  - что нужно переименовать в UI/API, чтобы убрать терминологическую путаницу.
- Сформировать test strategy для cleanup:
  - config migration tests;
  - API serialization tests;
  - settings page bootstrap tests;
  - runtime behavior tests для active templates и channel overrides.

## Предлагаемый план выпиливания

### Этап 1. Инвентаризация и маркировка deprecated-полей

- Добавить в документацию/код комментарии и diagnostics, где поле ещё считается legacy.
- Явно пометить:
  - `bot_settings.question_flow` как derived/deprecated;
  - `bot_settings.rating_system` как derived/deprecated;
  - `auto_close_hours` как migration fallback;
  - root `unblock_request_cooldown_minutes` как candidate for removal после usage audit.

### Этап 2. Перевод runtime на canonical reads

- Убедиться, что production reads идут только из:
  - `question_templates[active]`;
  - `rating_templates[active]`;
  - `auto_close_config.templates[active]`;
  - channel override columns.
- Legacy поля оставить только как fallback для старых конфигов.

### Этап 3. Перевод save-path на canonical writes

- Перестать писать новые user edits в legacy-поля как в самостоятельные сущности.
- Если compatibility ещё нужна, писать deprecated mirrors только на save boundary и только временно.

### Этап 4. Удаление channel JSON compatibility-ключей

- Вынести из `questions_cfg` всё, что уже имеет канонические колонки или отдельные settings payload.
- Оставить в `questions_cfg` только реально channel-local JSON, который ещё не вынесен в typed contract.

### Этап 5. Удаление frontend bridges

- Убрать fallback на:
  - `raw.question_flow`;
  - `raw.questions`;
  - `raw.rating_system`;
  - legacy `BOT_*` globals там, где runtime registry уже обязателен.
- Статус этапа:
  - fallback на `raw.question_flow`, `raw.questions`, `raw.rating_system` и legacy `BOT_*` globals для bot settings page уже убран;
  - remaining frontend/runtime cleanup сместился в channel compatibility и общий settings bootstrap contract.

### Этап 6. Финальная зачистка schema/config

- Мигрировать `config/shared/settings.json` и тестовые fixtures на чистую каноническую модель.
- Удалить deprecated reads, write-mirrors и устаревшие тесты/fixtures.

## Риски и ограничения

- Нельзя смешать cleanup этой задачи с полным удалением legacy dialogs modal/workspace fallback: это отдельный домен и отдельный rollout-риск.
- `questions_cfg` используется не только для template selection, но и для channel notification routing; нужна точная декомпозиция ключей перед удалением.
- `auto_close_hours` нельзя удалить в один шаг без проверки старых инсталляций и recovery/backfill сценариев.
- `dialog_config.question_templates` нельзя просто удалить как legacy, пока не принято решение, остаются ли operator question templates отдельной продуктовой сущностью.

## Критерии готовности

- Для каждого legacy/duplicate поля из settings-модели задокументирован статус: canonical / derived / deprecated / migration-only.
- Есть отдельный поэтапный план удаления legacy-слоя без скрытых источников истины.
- Понятно, какие поля удаляются, какие переименовываются, а какие остаются как отдельный домен.
- Сформирован список follow-up implementation tasks по backend, frontend, migrations и tests.

## Текущая точка остановки

- На текущем этапе settings page уже загружает bot settings через канонические `question_templates` / `rating_templates`, даже если raw shared config ещё хранит legacy `question_flow` / `rating_system`.
- Новые сохранения из UI уже идут в каноническом контракте, а legacy mirrors живут только на backend boundary как derived compatibility-слой.
- Shared config после нового сохранения больше не должен заново получать mirrors `bot_settings.question_flow`, `bot_settings.rating_system` и `auto_close_hours`, если был сохранён канонический payload.
- Legacy-аудит в UI и логах всё ещё помогает увидеть, что в shared config или channel runtime остаются старые поля.
- Dual-bootstrap страницы закрыт: следующий крупный риск теперь смещён в channel compatibility (`questions_cfg`) и в общие page-config fallback helper-ы.
- На channel API boundary legacy `question_template_id` / `rating_template_id` внутри `questions_cfg` уже перестают быть публичным контрактом: они поднимаются в typed response/save path и вычищаются из `questions_cfg`.
- На runtime-слое `java-bot` channel compatibility для выбора question/rating template тоже закрыт: `questions_cfg` больше не участвует в выборе активного шаблона, а legacy ключи используются только для явной диагностики миграции в логах.
- На client bootstrap settings page секционный payload (`dialog/admin/channels/parameters/appearance/locations`) стал единственным normal-path источником данных; legacy top-level aliases остаются только как compatibility outputs, но уже не читаются runtime-модулями страницы.
- `BotSettingsService` больше не использует root-level `question_flow` / `rating_system` как внутренний source of truth и больше не материализует их в sanitized maps.
- `MaintenanceTasks` всё ещё умеет читать legacy `auto_close_hours`, но только как read-only fallback для старых конфигов без `auto_close_config`; в canonical-конфигурации этот mirror больше не должен участвовать в runtime-решениях.
- Settings bootstrap/save boundary теперь уже приводит `auto_close_config.templates[*]` к canonical `hours`, а UI summary/editors больше не рассчитывают на `timeout_hours` / `auto_close_hours` внутри template payload.
- Runtime-боты (`telegram` / `vk` / `max`) уже работают через template-based helper API и больше не зависят от mirror-полей как от operational contract.
- `BotSettingsDto` больше не публикует root-level `question_flow` / `rating_system` в JSON-output; эти legacy keys остались только как compatibility input-path для чтения старых payload.
- Репозиторный canonical-config уже не содержит root-level `question_flow` / `rating_system`, так что оставшиеся mirrors существуют только как входная совместимость, а не как актуальный shared-config contract.
- Runtime по `auto_close_config.templates[*]` всё ещё принимает deprecated `timeout_hours` / `auto_close_hours`, но теперь делает это только как явно диагностируемый compatibility read-path.
- Следующая граница cleanup теперь сосредоточена в полном удалении read-fallback для top-level `auto_close_hours`, в аудите root `unblock_request_cooldown_minutes` и в остаточных expectations/fixtures вокруг legacy input-paths.
