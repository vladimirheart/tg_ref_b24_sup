# Задача [01-130]

## Короткое описание

Разделить giant dialogs.js на bounded client-side runtime modules.

## Цель

Убрать из `dialogs.js` роль client-side monolith и перевести runtime диалогов
на набор модулей с явными границами между list/details/workspace/actions/AI и
notification loops.

## Контекст

- По audit-срезу `dialogs.js` держит около `10426` строк.
- Внутри файла смешаны list/filter polling, details/history, workspace shell,
  quick actions, macro workflow, AI ops/review и bell refresh semantics.
- Backend домена `dialogs` уже существенно разрезан; следующий practical risk
  живёт в browser runtime, а не в `DialogService`.

## Что сделать

- Выделить shared client boundary для API helpers, polling timers, route/open
  state и refresh bus.
- Разрезать `dialogs.js` минимум на модули `list+filters`, `details+history`,
  `workspace shell`, `quick actions`, `macro workflow`, `AI ops/review`,
  `notifications refresh`.
- Убрать зависимости между соседними consumer surfaces через случайные global
  helpers там, где это возможно без переписывания API.
- Сохранить текущие runtime contracts между `/api/dialogs`, `details`,
  `history`, `workspace` и bell loop.

## Критерии готовности

- `dialogs.js` больше не является единственным giant runtime entrypoint.
- Основные dialog UI-сценарии разложены по bounded client-side модулям.
- Поведение quick actions, workspace, history и notifications сохраняется без
  регрессии.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog` и включить в него пользовательский промт, который привёл к изменению.
