# Задача [01-137]

## Короткое описание

Провести enterprise UI-проход Iguana в стиле Calm Operations без изменения продуктовой логики.

## Цель

Сделать основные рабочие экраны Iguana более строгими, плотными и операторскими: уменьшить декоративность, выровнять визуальные токены, усилить compact-first подачу и привести dialogs/dashboard/AI Ops/settings/knowledge к единому спокойному support-стилю.

## Контекст

- Пользователь попросил провести визуальную итерацию проекта **Iguana** в направлении `Enterprise support cockpit / Calm Operations`.
- Изменения должны затрагивать в первую очередь `spring-panel/src/main/resources/scss/`, `static/css/`, `templates/` и не должны ломать существующую продуктовую логику, маршруты, Thymeleaf-выражения и Bootstrap-поведение.
- В проекте уже есть штатный SCSS pipeline через `dart-sass-maven-plugin` в `spring-panel/pom.xml`, поэтому править нужно SCSS-источники и пересобрать публичные CSS-артефакты.
- В рабочем дереве уже есть несвязанные пользовательские изменения в `java-bot/bot-max/logs/bot-telegram.log`, `spring-panel/bot_database.db`, `spring-panel/settings.db`; их нельзя трогать.

## Что сделать

- Пересмотреть базовые визуальные токены и поверхности в SCSS: радиусы, тени, borders, плотность, семантические цвета, page-shell и panel language.
- Сделать sidebar строже и спокойнее, сохранив бренд Iguana и текущие состояния collapsed/expanded.
- Уплотнить и унифицировать страницы `dialogs`, `dashboard`, `dialogs/ai-ops`, `settings`, а также knowledge base, если она присутствует в проекте.
- Сохранить все текущие функции: compact mode, SLA, AI Ops, workspace, legacy fallback, export, settings governance, Bootstrap modal/collapse/dropdown.
- После изменений пересобрать SCSS -> CSS и прогнать проверки сборки/шаблонов.

## Критерии готовности

- Основные рабочие страницы выглядят заметно плотнее и спокойнее без радикальной смены архитектуры интерфейса.
- Dialogs воспринимается как operator-first inbox/workspace, AI Ops — как technical control center, settings — как строгий admin UI, knowledge base — как knowledge workspace.
- Изменения ограничены визуальным и UX-слоем и не ломают existing routes/controllers/Thymeleaf contracts.
- SCSS успешно пересобран в `static/css`, а проект проходит хотя бы базовые сборочные проверки.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog` и включить в него пользовательский промпт, который привёл к изменению.
