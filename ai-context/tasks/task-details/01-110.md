# Задача [01-110]

## Короткое описание

Добавить spam-блокировку клиента из модалки диалога и проверить blacklist/unblock сценарий.

## Цель

Дать оператору возможность пометить клиента как спам прямо из открытого диалога без перехода в карточку клиента, сохранив текущий permission-контур диалогов и не ломая существующие сценарии blacklist/unblock.

## Контекст

- Legacy-модалка диалога живёт в `spring-panel/src/main/resources/templates/dialogs/index.html` и `spring-panel/src/main/resources/static/js/dialogs.js`.
- Быстрые mutate-действия диалога проходят через `spring-panel/src/main/java/com/example/panel/controller/DialogQuickActionsController.java` и `spring-panel/src/main/java/com/example/panel/service/DialogQuickActionService.java`.
- Текущая работа с клиентским blacklist вынесена в `spring-panel/src/main/java/com/example/panel/controller/ClientBlacklistApiController.java`, но этот API завязан на `PAGE_CLIENTS`.
- Клиент подаёт запрос на снятие блокировки через `/unblock` в Telegram/VK ботах; оператор видит и обрабатывает такие запросы на `/unblock-requests`.
- В ходе проверки выявлена MAX-регрессия: запись клиента в `client_blacklist` из панели не останавливала входящие сообщения, потому что `java-bot/bot-max/src/main/java/com/example/supportbot/max/MaxWebhookController.java` не проверял `BlacklistService` на входе и не поддерживал self-service `/unblock`.
- Дополнительная live-проверка по `spring-panel/bot_database.db` подтвердила, что запись о блокировке для MAX-клиента `214991692` создаётся корректно (`is_blacklisted = 1`), но в момент пользовательской проверки уже после этой записи успел открыться новый тикет `34c9c836-6e34-4ad4-b450-a5c3e83db4bb`, то есть проблема была именно на стороне входной обработки `bot-max`, а не в панели.
- Текущий runtime `bot-max` держит `java-bot/bot-max/target/bot-max-0.0.1-SNAPSHOT.jar` открытым, поэтому `mvn clean` не может удалить артефакт; для применения фикса на стенде потребуется перезапуск процесса канала 2 после сборки.

## Что сделать

- Добавить отдельный quick action для пометки клиента как спам из модалки диалога с правами уровня `PAGE_DIALOGS`.
- Показать кнопку spam-блокировки рядом с действием `Закрыть` в legacy-модалке и подключить её к новому backend action.
- Проверить целостность сценария blacklist/unblock: постановка в blacklist, запрос на снятие, отображение у оператора, уведомления и заметные ограничения/риски.

## Критерии готовности

- Из модалки диалога можно заблокировать клиента как спам без перехода в раздел клиентов.
- Блокировка пишет причину и сохраняет историю так же, как существующий blacklist-flow.
- В итоговом отчёте зафиксировано, как клиент просит разблокировку и как это видит оператор, включая найденные ограничения.
