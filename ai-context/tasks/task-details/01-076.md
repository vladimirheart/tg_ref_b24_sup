# Задача [01-076]

## Короткое описание

Перевести `iikoServer` креды локаций на явный `SHA-1` пароль и убрать двусмысленность `API secret`.

## Цель

Сделать credential contract для источников `iikoServer` однозначным: оператор должен задавать
именно `SHA-1` пароль пользователя `iikoServer`, а не путать его с transport-style `API secret`.

## Контекст

- Для `iikoServer` и для transport API используются разные auth-модели, но исторически UI и
  тексты могли создавать впечатление, что это один и тот же тип секрета.
- В текущем settings UI уже видны признаки нужной модели: отдельное поле `SHA-1 пароль` и
  подсказка про `40` hex-символов.
- В service-слое уже есть валидация такого формата в
  `LocationsIikoServerSourceSettingsService`, поэтому задача фиксирует и закрепляет boundary на
  уровне UX, storage и runtime semantics.
- Для `Track A3` это базовый compatibility contract, который нельзя размыть при future split.

## Пересечения с backlog

- `01-072` выделяет editor источников; `01-076` уточняет в нём семантику секретных полей.
- `01-075` зависит от корректной credential model для авторизации ручного sync.
- `01-108` остаётся отдельной задачей про transport/base URL и не должен снова втягивать
  `iikoServer` в `api secret`-терминологию.

## Что сделать

- Закрепить в UI, DTO и persistence явную модель `SHA-1 password` для `iikoServer`-источников.
- Убрать двусмысленные названия и подсказки вроде `api secret`, если они относятся именно к
  location-source сценарию.
- Сохранить секретное значение как secret field с корректным поведением “оставить пустым, чтобы не менять”.
- Проверить, что downstream sync/runtime используют именно эту credential model.

## Критерии готовности

- Для `iikoServer`-источников в UI и backend используется явный `SHA-1` password contract.
- Пользователь больше не путает location-source credential с transport `API secret`.
- Secret field работает предсказуемо: сохранённый пароль не раскрывается обратно, но может быть заменён.
