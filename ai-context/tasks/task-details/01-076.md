# Задача [01-076]

Перевести настройки источников `iikoServer` для локаций на явный `SHA-1` пароль вместо абстрактного `API secret`.

## Цель

Сделать контракт настроек прозрачным:

- пользователь вводит `login`;
- пользователь вводит `пароль в формате SHA-1`;
- система использует это значение как `pass` в `GET /resto/api/auth`;
- backend валидирует формат и не делает неочевидного автопреобразования.

## Контекст

- После исправления endpoint ручная sync больше не падает на `404`, но пользователь справедливо указал, что поле названо неудачно и вводимый секрет должен быть именно `SHA-1` паролем.
- Текущая реализация ещё умеет автоматически хэшировать обычный текст в `SHA-1`, что создаёт двусмысленность и мешает понимать, что именно сохранено в настройках.

## Что сделать

- В UI переименовать поле `API secret` в `SHA-1 пароль`.
- Добавить пояснение, что нужно вводить уже готовый 40-символьный hex `SHA-1`.
- В backend валидировать формат `SHA-1` для сохранённого значения.
- Убрать автозахэширование в runtime gateway и использовать переданное `SHA-1` значение как есть.
- Обновить/добавить тесты на:
  - использование переданного `SHA-1` без преобразования;
  - отказ сохранения при невалидном формате.

## Критерии готовности

- В настройках источников локаций больше нет формулировки `API secret`.
- Невалидный пароль не сохраняется и отдаёт понятную ошибку.
- Runtime auth использует только уже введённый `SHA-1` пароль.
