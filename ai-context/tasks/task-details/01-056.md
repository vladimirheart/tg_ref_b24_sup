# Задача [01-056]

## Короткое описание

Перевести RMS UI на вычисляемое `display`-имя вместо сырого `server_name` и добавить временную шкалу состояний в диагностические представления мониторингов.

## Цель

Убрать ситуацию, когда в `/analytics/rms-control` пользователю показывается сырое или испорченное имя RMS, даже если его можно восстановить из доступных данных, и дать в диагностике не только последний снимок, но и историю состояний по времени.

## Контекст

- Пользователь сообщил, что для двух RMS (`bb-bryansk-trc-bum-fk.iiko.it` и `bb-baumana.iiko.it`) имя на странице всё ещё отображается криво.
- Проверка `monitoring.db` показала, что в хранилище эти записи уже могут лежать корректно, поэтому UI не должен жёстко зависеть только от сырого `server_name`.
- Для RMS и iiko уже есть отдельные диагностические/детальные модалки, но в них нет временной шкалы смены состояний.
- Пользователь отдельно попросил зафиксировать это как правило проекта.

## Что сделать

- Добавить вычисляемое `display`-имя RMS в backend-ответы и использовать его в UI.
- Ввести общий history-слой в `monitoring.db` для результатов проверок мониторингов.
- Показывать последние события по временной шкале в RMS diagnostics и iiko diagnostics.
- Добавить repo-specific правило в `ai-context/rules`, что мониторинговая диагностика должна показывать историю состояний, а не только последнее состояние.

## Критерии готовности

- `mvnw -q -DskipTests compile` проходит.
- `/analytics/rms-control` использует вычисляемое display-имя RMS.
- В RMS diagnostics и iiko diagnostics есть временная шкала последних проверок.
- В `ai-context/rules` добавлено отдельное правило по diagnostics history для мониторингов.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog`.
