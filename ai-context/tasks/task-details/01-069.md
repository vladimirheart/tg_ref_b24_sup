# Задача [01-069]

Сделать запуск `spring-panel` в Windows устойчивым к падению `Maven clean`, когда `target` заблокирован локальным процессом.

## Цель

Убрать ситуацию, в которой `run-windows.bat` полностью прекращает запуск панели из-за неуспешного `clean`, хотя проект можно продолжить собирать и запускать без этой фазы.

## Контекст

- `spring-panel/run-windows.bat` принудительно запускал `clean spring-boot:run` одной Maven-командой.
- На Windows каталог `spring-panel/target` может быть заблокирован IDE, Java Language Server или предыдущим Java-процессом.
- `run-linux.sh` не требует обязательного `clean`, поэтому Windows-скрипт был менее устойчивым к локальным файловым локам.

## Что сделать

- Разделить `clean` и `spring-boot:run` в `run-windows.bat`.
- Оставить `clean` best-effort фазой: при ошибке выводить предупреждение и всё равно продолжать запуск панели.
- Добавить явную переменную `SPRING_PANEL_SKIP_CLEAN`, чтобы пользователь мог отключить clean без правки скрипта.

## Критерии готовности

- Если `mvn clean` падает на удалении `target`, `run-windows.bat` не завершает работу сразу и пробует `spring-boot:run`.
- Пользователь видит понятное предупреждение о возможной блокировке файлов в `target`.
- Скрипт сохраняет возможность принудительно пропустить clean через `SPRING_PANEL_SKIP_CLEAN`.
