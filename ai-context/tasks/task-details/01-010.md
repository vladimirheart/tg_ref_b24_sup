# Задача [01-010]

## Короткое описание

Довести MAX-бота до рабочей webhook-интеграции через `spring-panel`.

## Цель

Сделать так, чтобы MAX-бот корректно работал end-to-end:

- поднимался как web-приложение;
- принимал webhook на публичном origin панели;
- маршрутизировался в правильный внутренний бот-процесс;
- показывал в UI и документации реальный, а не фиктивный webhook URL.

## Контекст и ограничения

- MAX по официальной документации использует webhook для production и long
  polling для development/test.
- В текущем проекте MAX-бот запускается отдельным процессом `java-bot/bot-max`.
- Панель ранее показывала URL `.../webhooks/max`, но сама такой endpoint не
  обслуживала.
- Для долгоживущего решения добавлено project-specific правило в
  `ai-context/rules/backend`.

## Что сделано

- MAX-процесс теперь запускается как локальный servlet-сервис на отдельном порту
  `127.0.0.1`.
- Публичный webhook перенесён на сторону `spring-panel` в маршрут
  `/webhooks/max/{channelId}` с проксированием во внутренний процесс `bot-max`.
- В `SecurityConfig` разрешён анонимный доступ к MAX webhook и отключён CSRF
  для этого пути.
- UI и документация обновлены под channel-specific webhook URL.
- Добавлено repo-specific правило о маршрутизации webhook-платформ через panel.
