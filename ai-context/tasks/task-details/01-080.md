# Задача [01-080]

Добавить `MTProto` в создание прокси-маршрутов и профилей, не симулируя поддержку там, где текущий runtime её не умеет.

## Цель

Расширить настройки сетевых маршрутов для ботов и проекта:

- пользователь может выбрать `MTProto` при создании прокси-профиля;
- пользователь может задать `host`, `port` и `secret`;
- система хранит и показывает эти данные корректно;
- текущий Telegram bot runtime не пытается трактовать `MTProto` как обычный `HTTP/SOCKS` прокси и явно сообщает об ограничении.

## Контекст

- Сейчас настройки маршрутов и прокси-профилей поддерживают `http`, `https`, `socks4`, `socks5` и `vless`.
- Для `VLESS` уже существует особый контракт с отдельным `token`, поэтому `MTProto` логично вводить как ещё один special-case тип.
- Текущий Telegram-бот в проекте использует Bot API по HTTPS через `telegrambots` и `DefaultBotOptions`, а не прямой MTProto transport.
- Значит, добавить `MTProto` как “полностью рабочий транспорт” без внешнего адаптера было бы технически некорректно.

## Что сделать

- Добавить `mtproto` в backend-нормализацию прокси-схем.
- Расширить модель прокси отдельным полем `secret`.
- Обновить UI форм маршрутов и прокси-профилей:
  - новый пункт `MTProto`;
  - отдельное поле `secret`;
  - корректные подсказки и валидация.
- Скорректировать runtime-контракт:
  - сохранять `MTProto` в конфигурации;
  - не маппить его в текущие `HTTP_PROXY` / `ALL_PROXY` / Java proxy options;
  - при попытке запуска Telegram-бота с `MTProto` возвращать явную диагностическую ошибку.
- Обновить тесты для network/service/runtime-контрактов.

## Критерии готовности

- В UI можно создать профиль и маршрут с типом `MTProto`.
- Для `MTProto` используется поле `secret`, а не `token`.
- Telegram runtime не делает вид, что умеет запускаться через `MTProto`, и сообщает об этом явно.
- Тесты покрывают новый тип схемы и защиту от неверного запуска.
