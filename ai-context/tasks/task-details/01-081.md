# Задача [01-081]

Убрать `MTProto` из настроек маршрутов и runtime, так как в текущей архитектуре Telegram Bot API он не даёт рабочего результата.

## Цель

Вернуть настройки сети к честному и рабочему контракту:

- в UI остаются только реально поддерживаемые режимы проксирования;
- в backend и runtime не остаётся мёртвого special-case для `mtproto`;
- пользователь не может выбрать режим, который не работает напрямую для бота.

## Контекст

- После внедрения `MTProto` выяснилось, что в рамках текущего Java runtime и Bot API over HTTPS это нерабочий пользовательский режим.
- Пользователь прямо подтвердил, что такую реализацию нужно убрать.
- История изменений и changelog по `01-080` остаются как есть по append-only правилу.

## Что сделать

- Убрать `MTProto` из форм маршрутов и профилей в settings UI.
- Удалить поля и валидацию `secret`, связанные только с `MTProto`.
- Убрать `mtproto` из backend-нормализации схем прокси.
- Удалить runtime fallback/warning-ветки, добавленные только для `MTProto`.
- Обновить тесты под итоговый поддерживаемый набор схем.

## Критерии готовности

- В UI больше нельзя выбрать `MTProto`.
- В backend и runtime нет отдельного `mtproto`-кода.
- Целевые тесты и сборка проходят.

## Дополнительные условия

- После file changes создать отдельный changelog в `ai-context/changelog`.
- В финальном ответе подготовить `Резюме для коммита`.
