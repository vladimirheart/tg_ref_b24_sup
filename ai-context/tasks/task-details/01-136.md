# Задача [01-136]

## Короткое описание

Починить server-side navbar/template contract страницы `dialogs` для
зелёного `DialogsControllerWebMvcTest`.

## Цель

- вернуть page-template `/dialogs` в устойчивое server-rendered состояние,
  где `dialogs/index` и подключаемые fragment'ы не падают из-за отсутствующих
  model flags;
- довести `DialogsControllerWebMvcTest` до зелёного статуса как отдельный
  backend/template follow-up, не смешивая его с уже завершённым client-side
  runtime split `01-130`;
- зафиксировать явный contract между `DialogsController`,
  `NavigationService` и `fragments/navbar.html`, чтобы page smoke для
  `/dialogs`, `/dialogs/{ticketId}` и `/ai-ops` не зависел от неявного
  side-effect поведения mock/stub окружения.

## Контекст

- после закрытия `01-130` runtime regressions страницы диалогов зелёными
  подтверждены на integration/test-net уровне, а remaining red path остался в
  `DialogsControllerWebMvcTest`;
- текущий красный симптом не относится к `dialogs.js` и вынесенным runtime
  модулям: `fragments/navbar.html` ожидает model-флаги
  `canManageChannels`, `canViewSettings` и соседние permission attributes;
- в production эти атрибуты сейчас наполняет `NavigationService.enrich(...)`,
  но в `DialogsControllerWebMvcTest` сервис замокан целиком, поэтому
  fragment-side contract оказывается хрупким и может ломать Thymeleaf
  rendering;
- задача должна решить именно server-side contract problem: либо через более
  устойчивое обогащение model, либо через безопасный fragment/template
  fallback, либо через явную test contract setup, но без отката bounded
  dialogs runtime обратно в giant `dialogs.js`.

## Что проверить

- какие именно navbar/head fragment attributes обязательны для рендера
  `dialogs/index` и `dialogs/ai-ops`;
- должен ли `DialogsController` сам гарантировать минимальный набор
  permission/model flags, даже если `NavigationService` замокан или частично
  недоступен;
- можно ли сделать `fragments/navbar.html` устойчивым к отсутствующим model
  keys без размывания permission semantics;
- не страдают ли тем же contract gap другие page-level `WebMvcTest`, где
  navigation enrichment тоже подменяется mock'ом.

## Что сделать

- Локализовать точную причину падения `DialogsControllerWebMvcTest` на уровне
  `DialogsController` / `NavigationService` / `fragments/navbar.html`.
- Выбрать и реализовать один устойчивый contract fix path для server-rendered
  `/dialogs` страниц и соседнего `ai-ops` page template.
- Обновить или расширить `DialogsControllerWebMvcTest`, чтобы он явно
  закреплял нужный navbar/template contract и не маскировал регрессию.
- При необходимости проверить соседние template smoke/controller tests,
  если выбранный fix path затрагивает общий navigation fragment contract.

## Критерии готовности

- `DialogsControllerWebMvcTest` проходит зелёным целиком.
- `/dialogs`, `/dialogs/{ticketId}` и `/ai-ops` продолжают рендериться через
  те же templates без падения на navbar/head fragment contract.
- Исправление остаётся в server-side controller/template области и не
  затрагивает уже закрытый bounded runtime split из `01-130`.
