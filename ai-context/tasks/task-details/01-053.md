# Задача [01-053]

## Короткое описание

Починить поиск лицензии RMS `RMS (Front Fast Food)` с `id=100` и сделать
диагностику RMS в UI читаемой без битой кодировки.

## Цель

Добиться, чтобы мониторинг RMS корректно находил целевую лицензию даже при
другой структуре XML-ответа, а диагностический вывод в UI отображался с
нормальными переносами строк и без mojibake.

## Контекст

- Текущая логика RMS ищет целевую лицензию по ограниченному набору ключей и
  ориентируется в основном на дочерние XML-узлы.
- В реальном ответе RMS `id` и часть метаданных могут приходить в атрибутах
  XML-элементов или в других ключах.
- Диагностические блоки `ping`, `tracert` и raw licensing response должны быть
  читабельны прямо из UI на странице `/analytics/rms-control`.
- Правила task-flow и changelog описаны в
  `C:\Users\SinicinVV\git_h\tg_ref_b24_sup\ai-context\baseline\guides\tasks.md`.

## Что сделать

- Расширить парсинг лицензий RMS на XML-атрибуты и более гибкий матч целевой
  лицензии `RMS (Front Fast Food)` с `id=100`.
- Нормализовать декодирование и форматирование диагностического вывода для
  licensing response, `ping` и `tracert`.
- Обновить UI модалки диагностики RMS, чтобы длинный текст и структурированные
  блоки читались без слитного вывода.

## Критерии готовности

- Статус лицензии RMS больше не падает в ложную ошибку "Не найдена лицензия ..."
  в случаях, когда лицензия `id=100` реально присутствует в ответе.
- Диагностика RMS в UI показывает многострочный и читаемый текст.
- Проект успешно проходит `mvnw -q -DskipTests compile`.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из
  `ai-context/baseline/guides/tasks.md`.
- После любых file changes создать отдельный changelog-файл в
  `ai-context/changelog` по правилу «одно изменение = один файл».
