# Задача [01-129]

## Короткое описание

Разрезать giant settings runtime на bounded client-side модули.

## Цель

Превратить giant inline/settings runtime в набор более узких клиентских
модулей по предметным subdomains, чтобы дальнейшая поддержка `settings` не
упиралась в один page-script и один giant template.

## Контекст

- После `01-128` page-shell bootstrap должен быть отделён от внутренней
  предметной логики settings.
- По audit-карте внутри settings выделены отдельные sub-tracks:
  `parameters/legal entities/partner contacts/IT catalog`,
  `client statuses/business styles`,
  `locations tree/sync/wizard`,
  `dialog templates/auto-close/SLA/workspace governance`,
  `channels/network routes/runtime status/editor`,
  `reporting/manager bindings/modal shell`.
- Эта задача не про новый backend split, а про client-side decomposition и
  явные runtime boundaries.
- Для части subdomains уже есть отдельный исторический backlog, который нельзя
  считать “с нуля”: locations-пакеты `01-071`..`01-077`, channel/settings UX
  пакеты `01-112`..`01-115`, а также channel editor extension из `01-121`.

## Пересечения с backlog

- `01-071`..`01-077` — locations sync и iikoServer runtime уже считаются
  предметным baseline для location/runtime subtrack.
- `01-112`..`01-115` — channels/dialog templates/modal UX уже были локально
  доведены и должны только сохраняться при module split.
- `01-121` — channel working hours уже встроены в settings runtime и должны
  уехать в новый module split без потери поведения.

## Что сделать

- Разбить giant settings runtime на отдельные JS-модули по bounded subdomains.
- Перевести template с прямых inline entry-points на `data-*` hooks и
  импортируемые runtime initializers.
- Уменьшить зависимость subdomains друг от друга через глобальные функции и
  shared mutable state.
- Зафиксировать migration order так, чтобы сначала переезжали наиболее
  изолированные subdomains, а history-heavy `locations` и `channels` шли уже на
  готовом page-shell baseline после `01-128`.
- Сохранить текущие пользовательские сценарии `parameters`, `locations`,
  `dialog settings`, `channels` и `reporting`.

## Критерии готовности

- Основные settings subdomains больше не живут в одном giant runtime-скрипте.
- Для каждого крупного subdomain есть явная точка инициализации и ownership.
- Template `settings/index.html` перестаёт быть единственным местом, где
  собрана почти вся клиентская логика настроек.

## Текущий статус

- Статус задачи: `🟡 в работе`.
- По состоянию на `2026-06-19` задача идёт пакетным runtime-split поверх
  baseline из `01-128`: значимая часть settings subdomains уже вынесена в
  отдельные JS-модули, но `settings/index.html` всё ещё держит остатки
  channel persistence flow и часть legacy helpers.
- После удаления public form из проекта channel editor больше не должен
  держать мёртвые `publicform/public_id/questions_cfg.fields` хвосты в
  template и runtime-модулях.
