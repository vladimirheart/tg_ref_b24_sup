# Задача [01-024]

## Заголовок

Начать поэтапный рефакторинг архитектуры и UI runtime по итогам аудита

## Контекст

По результатам аудита в проекте есть два связанных класса проблем:

- архитектурная перегрузка в крупных доменах `dialogs` и `settings`;
- фрагментированный UI runtime, где тема, presets и page bootstrap подключаются
  неравномерно.

В рамках первого этапа нужно не разрезать весь проект целиком, а заложить
foundation-слой, который уменьшит расхождение между страницами и создаст основу
для дальнейшего безопасного рефакторинга.

## Что нужно сделать

- описать пошаговый roadmap рефакторинга архитектуры и UI;
- централизовать ранний head-bootstrap для темы и `ui-config`;
- расширить `ui-config` на основные разделы панели;
- ввести единый runtime-модуль для browser-only UI preferences;
- перевести на него хотя бы `theme` и `sidebar`, чтобы сократить прямое
  использование `localStorage`;
- добавить server-backed слой для operator UI preferences и раннюю гидрацию
  prefs в `head`;
- перевести operator triage preferences диалогов из shared JSON в server-backed
  UI preferences с backward-compatible fallback;
- начать реальный read/write split `dialogs` с вынесением triage preference
  сценария из `DialogApiController` в отдельный сервис;
- завершить endpoint split triage preferences диалогов и вынести сам API-слой
  в отдельный controller;
- начать read-only split `dialogs` с вынесением list endpoint и SLA
  orchestration в отдельные controller/service;
- продолжить read-only split `dialogs`: вынести details/history/public metrics
  endpoints в отдельный controller/service слой;
- вынести quick actions диалогов (`reply/edit/delete/media/resolve/reopen/categories/take/snooze`)
  из `DialogApiController` в отдельные controller/service;
- вынести macro API диалогов (`/macro/dry-run`, `/macro/variables`) в
  отдельные controller/service;
- вынести workspace telemetry API и summary orchestration в отдельные
  controller/service;
- вынести AI ops API диалогов (`ai-suggestions`, `ai-control`, `ai-review`,
  `ai-solution-memory`, `ai-monitoring`) в отдельные controller/service;
- выделить общий authorization/service слой для dialog permission checks,
  чтобы новые dialog controllers не дублировали проверки и audit logging;
- перевести `DialogQuickActionsController` и remaining workspace permission flow
  на общий `DialogAuthorizationService`, чтобы убрать локальные permission
  helper'ы из controller-слоя;
- вынести оставшийся workspace endpoint диалогов и его orchestration в
  отдельные `DialogWorkspaceController` и `DialogWorkspaceService`, чтобы
  завершить controller-level split домена `dialogs`;
- вынести macro template governance/normalization из
  `SettingsBridgeController` в отдельный `SettingsMacroTemplateService`;
- вынести основной update-use-case `/settings` из
  `SettingsBridgeController` в отдельный `SettingsUpdateService`, оставив в
  controller только thin-wrapper;
- вынести top-level settings (`auto_close`, `categories`, `client_statuses`,
  `integration/reporting` и похожие базовые ключи) из `SettingsUpdateService`
  в отдельный `SettingsTopLevelUpdateService`;
- вынести обработку `locations` и связанный parameter sync из
  `SettingsUpdateService` в отдельный `SettingsLocationsUpdateService`;
- вынести giant `dialog_config` update flow из `SettingsUpdateService` в
  отдельный `SettingsDialogConfigUpdateService`, оставив в
  `SettingsUpdateService` только orchestration;
- разрезать giant `SettingsDialogConfigUpdateService` по поддоменам хотя бы на
  `SLA/AI`, `workspace`, `public form` и общий support/router слой;
- начать `settings` domain split с выделением parameters API и связанной
  location-sync логики из `SettingsBridgeController`;
- вынести `it-equipment` API из `SettingsBridgeController` в отдельные
  controller/service с сохранением уведомлений;
- убрать позднюю загрузку `theme.js` из sidebar-fragment;
- сохранить рабочее текущее поведение страниц без массового визуального
  редизайна.

## Критерии готовности

- есть отдельный roadmap-файл в `docs`;
- `theme.js` и `ui-config.js` подключаются через единый head-fragment;
- browser-only UI preferences читаются через общий runtime-модуль, а не через
  хаотичные прямые обращения минимум в базовых shared-скриптах;
- для authenticated user есть server-backed bootstrap/sync слой UI preferences;
- triage preferences оператора больше не завязаны только на
  `settings.json/dialog_config`, а имеют server-backed источник правды;
- route списка диалогов больше не живёт в монолитном `DialogApiController`, а
  вынесен в отдельный controller/service слой;
- details/history/public metrics endpoints диалогов больше не живут только в
  `DialogApiController`, а собраны в отдельном read-layer;
- triage preferences диалогов больше не обслуживаются `DialogApiController`, а
  вынесены в отдельный controller поверх уже выделенного сервиса;
- quick actions диалогов больше не живут в `DialogApiController`, а вынесены в
  отдельный controller/service слой;
- macro endpoints диалогов больше не живут в `DialogApiController`, а вынесены
  в отдельный controller/service слой;
- workspace telemetry endpoints больше не живут в `DialogApiController`, а
  вынесены в отдельный controller/service слой;
- AI ops endpoints больше не живут в `DialogApiController`, а вынесены в
  отдельный controller/service слой;
- dialog permission checks и audit logging больше не копируются по новым
  controller-слоям, а имеют выделенный shared service;
- `DialogQuickActionsController` больше не содержит собственных dialog
  permission helper'ов и использует общий `DialogAuthorizationService`;
- workspace endpoint диалогов больше не живёт в controller-монолите, а вынесен
  в отдельные `DialogWorkspaceController` и `DialogWorkspaceService`;
- macro template governance для настроек диалогов больше не живёт внутри
  `SettingsBridgeController`, а вынесен в отдельный `SettingsMacroTemplateService`;
- основной update-use-case `/settings` больше не реализован внутри
  `SettingsBridgeController`, а делегирован в отдельный `SettingsUpdateService`;
- базовые top-level settings больше не собираются прямо в
  `SettingsUpdateService`, а вынесены в `SettingsTopLevelUpdateService`;
- `locations` save/sync больше не реализуются прямо в `SettingsUpdateService`,
  а вынесены в `SettingsLocationsUpdateService`;
- giant `dialog_config` block больше не живёт внутри `SettingsUpdateService`,
  а вынесен в `SettingsDialogConfigUpdateService`;
- `SettingsDialogConfigUpdateService` больше не является единственной giant
  точкой обновления `dialog_config`, а делегирует хотя бы в `SLA/AI`,
  `workspace`, `public form` и support/router services;
- `/api/settings/parameters` больше не живёт внутри `SettingsBridgeController`,
  а выделен в отдельные controller/service с сохранением location sync;
- `/api/settings/it-equipment` больше не живёт внутри `SettingsBridgeController`,
  а выделен в отдельные controller/service с сохранением notification flow;
- страницы, использующие базовый UI-слой, больше не подключают `ui-config`
  разрозненно;
- `ui-config` покрывает не только `dashboard/settings/dialogs`, но и остальные
  основные разделы;
- задача переведена в `🟣` после завершения AI.
