# Задача [01-024]

## Заголовок

Начать поэтапный рефакторинг архитектуры и UI runtime по итогам аудита

## Контекст

По результатам аудита в проекте есть два связанных класса проблем:

- архитектурная перегрузка в крупных доменах `dialogs` и `settings`;
- фрагментированный UI runtime, где тема, presets и page bootstrap подключаются
  неравномерно.

В рамках первого этапа нужно не разрезать весь проект целиком, а заложить
foundation-слой, который уменьшит расхождение между страницами и создаст основу
для дальнейшего безопасного рефакторинга.

## Что нужно сделать

- описать пошаговый roadmap рефакторинга архитектуры и UI;
- централизовать ранний head-bootstrap для темы и `ui-config`;
- расширить `ui-config` на основные разделы панели;
- ввести единый runtime-модуль для browser-only UI preferences;
- перевести на него хотя бы `theme` и `sidebar`, чтобы сократить прямое
  использование `localStorage`;
- добавить server-backed слой для operator UI preferences и раннюю гидрацию
  prefs в `head`;
- перевести operator triage preferences диалогов из shared JSON в server-backed
  UI preferences с backward-compatible fallback;
- начать реальный read/write split `dialogs` с вынесением triage preference
  сценария из `DialogApiController` в отдельный сервис;
- начать read-only split `dialogs` с вынесением list endpoint и SLA
  orchestration в отдельные controller/service;
- убрать позднюю загрузку `theme.js` из sidebar-fragment;
- сохранить рабочее текущее поведение страниц без массового визуального
  редизайна.

## Критерии готовности

- есть отдельный roadmap-файл в `docs`;
- `theme.js` и `ui-config.js` подключаются через единый head-fragment;
- browser-only UI preferences читаются через общий runtime-модуль, а не через
  хаотичные прямые обращения минимум в базовых shared-скриптах;
- для authenticated user есть server-backed bootstrap/sync слой UI preferences;
- triage preferences оператора больше не завязаны только на
  `settings.json/dialog_config`, а имеют server-backed источник правды;
- route списка диалогов больше не живёт в монолитном `DialogApiController`, а
  вынесен в отдельный controller/service слой;
- страницы, использующие базовый UI-слой, больше не подключают `ui-config`
  разрозненно;
- `ui-config` покрывает не только `dashboard/settings/dialogs`, но и остальные
  основные разделы;
- задача переведена в `🟣` после завершения AI.
