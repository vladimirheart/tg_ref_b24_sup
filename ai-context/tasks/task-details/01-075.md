# Задача [01-075]

Починить авторизацию `iikoServer API` в live-синхронизации департаментов, потому что ручной запуск падает с `HTTP 404 при получении access token`.

## Цель

Вернуть рабочую синхронизацию департаментов из `iikoServer API` для настроенных chain-серверов без смешения с `iiko biz/transport` авторизацией.

## Контекст

- Сейчас `IikoDepartmentLocationCatalogService` получает департаменты через `iikoServer API`.
- При этом auth в коде реализован через endpoint `/api/0/auth/access_token` с параметрами `user_id/user_secret`, что характерно для другого контура API.
- Реальные chain-серверы пользователя (`gk-sv.iiko.it`, `bb-chain.iiko.it`) отвечают на этот path `404`, а на `GET /resto/api/auth` и `GET /resto/api/corporation/departments/` отдают осмысленные `401`, что указывает на корректный server-api base path.

## Что сделать

- Перевести auth на `iikoServer API` endpoint `GET /resto/api/auth`.
- Использовать корректные query-параметры `login` и `pass`.
- Не ломать текущие сохранённые секреты:
  - если введён уже готовый `SHA-1`, использовать его как есть;
  - если введён обычный секрет, пробовать `SHA-1` по документации и fallback на raw-значение для совместимости.
- Добавить тест, который проверяет реальный HTTP flow auth + departments через локальный test server.

## Критерии готовности

- Ручная sync больше не обращается к `/api/0/auth/access_token`.
- Запросы к `iikoServer` идут через `/resto/api/auth` и `/resto/api/corporation/departments/`.
- Есть автотест, который фиксирует правильный auth path и query names.
