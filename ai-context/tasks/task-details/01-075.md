# Задача [01-075]

## Короткое описание

Починить авторизацию `iikoServer API` для ручной синхронизации департаментов.

## Цель

Сделать ручной sync департаментов реально рабочим при валидных `iikoServer`-кредах, чтобы
оператор мог запустить синхронизацию из UI без скрытых auth-сбоев.

## Контекст

- После появления ручного sync (`01-074`) главным практическим риском становится не только UI,
  но и реальная авторизация в `iikoServer API`.
- В settings-контуре для `iikoServer` уже выделены свои source settings, а в service-слое есть
  отдельные компоненты для их чтения и выполнения sync.
- Задача относится к bounded runtime-поведению location sync, а не к общему transport API flow.
- Её нельзя “чинить” ценой повторного смешения `iikoServer`-кредов с `iiko biz` /
  `api secret` transport-профилями.

## Пересечения с backlog

- `01-072` задаёт отдельный settings editor для `iikoServer`-источников.
- `01-074` зависит от того, что ручной запуск умеет проходить auth и не висит в ложном progress.
- `01-076` уточняет credential contract, поэтому auth fix должен быть совместим с явной
  моделью `SHA-1 password`.

## Что сделать

- Разобрать и исправить auth flow ручной синхронизации against `iikoServer API`.
- Убедиться, что sync использует корректные поля источника и не подменяет их transport-кредами.
- Отдавать в UI понятную ошибку авторизации, если креды реально невалидны.
- Закрыть regression net на успешный и неуспешный auth path.

## Критерии готовности

- При валидных `iikoServer`-кредах ручной sync проходит авторизацию и получает данные департаментов.
- При невалидных кредах UI видит понятную auth-ошибку, а не молчаливый сбой.
- Ручной sync не зависит от `iiko biz` / transport credential flow.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из
  `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog` и включить в него
  пользовательский промпт, который привёл к изменению.
