# Задача [01-031]

## Заголовок

Добавить decision trace API, offline eval-контур и расширенный KPI-мониторинг AI-агента

## Контекст

Без наблюдаемости и offline-набора сложно корректно подбирать thresholds и
проверять реальное качество AI-решений. Нужен измеримый контур для инженерной
и операционной отладки.

## Что нужно сделать

- добавить API для отладки:
  `GET /api/dialogs/{ticketId}/ai-decision-trace`,
  `POST /api/dialogs/{ticketId}/ai-reclassify`,
  `POST /api/dialogs/{ticketId}/ai-retrieve-debug`,
  `GET/POST /api/dialogs/ai-intents`,
  `GET/POST /api/dialogs/ai-knowledge-units`;
- расширить event/state trace полями policy/source/threshold/intent/sensitive;
- ввести offline eval set (минимум 300 кейсов) и процедуру регулярного прогона;
- добавить KPI:
  `wrong_auto_reply_rate`, `auto_reply_from_untrusted_source_rate`,
  `sensitive_topic_block_rate`, `approved_memory_reuse_rate`,
  `stale_memory_hit_rate`, `review_queue_age_p95`, `review_queue_growth_rate`.

## Критерии готовности

- для любого диалога можно получить decision trace без чтения сырых логов;
- оффлайн-набор данных существует и используется в регулярной оценке;
- KPI доступны в мониторинге и используются для управленческих решений;
- изменение thresholds опирается на данные, а не на ручные догадки.
