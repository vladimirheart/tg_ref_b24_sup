# Задача [01-031]

## Короткое описание

Добавить decision trace API, offline eval-контур и расширенный KPI-мониторинг AI-агента.

## Цель

Зафиксировать и довести единый operational-контур AI-агента, в котором оператор
может просматривать decision trace по диалогу, запускать и анализировать
offline evaluation, а также наблюдать ключевые KPI и алерты в AI Ops без
обращения к сырым логам.

## Контекст

- В репозитории уже есть существенная часть целевого контура:
  `DialogAiOpsController`, `DialogAiOpsService`, `AiOfflineEvaluationService`,
  UI-страница `dialogs/ai-ops.html`, runtime `ai-ops.js` и тесты вокруг
  `DialogAiOpsControllerWebMvcTest` и `AiOfflineEvaluationServiceTest`.
- Для decision trace уже используется API
  `GET /api/dialogs/{ticketId}/ai-decision-trace`.
- Для offline eval уже заведены API
  `GET /api/dialogs/ai-monitoring/offline-eval` и
  `POST /api/dialogs/ai-monitoring/offline-eval/run`, а dataset хранится в
  `spring-panel/src/main/resources/ai/offline-eval-templates.json`.
- AI Ops уже показывает базовые KPI (`auto_reply_rate`, `assist_usage_rate`,
  `escalation_rate`, `correction_rate`), алерты, runbook и recent events; эту
  задачу нужно вести как единый слой observability/evaluation для AI-агента.
- Архитектурный контекст и ожидания по этому слою уже упомянуты в
  `docs/ai_dialog_agent_plan.md`,
  `docs/ARCHITECTURE_AUDIT_2026-04-08.md` и
  `docs/ARCH_UI_REFACTORING_ROADMAP_2026-04-15.md`.
- У задачи не было собственного `task-details` файла, хотя продуктовый и
  тестовый контур в коде уже присутствует, поэтому при дальнейшем выполнении
  важно опираться на фактическое состояние репозитория, а не на предположения.

## Что сделать

- Проверить и при необходимости довести decision trace API до устойчивого
  контракта для AI Ops и диагностики конкретного диалога.
- Проверить и при необходимости довести offline eval-контур:
  dataset, ручной запуск, плановый запуск, сохранение последнего run и
  диагностический payload по ошибкам/coverage.
- Проверить и при необходимости довести расширенный KPI-мониторинг AI-агента:
  summary-метрики, alerts, runbook, recent events и связанный UI AI Ops.
- Синхронизировать тестовое покрытие и task/doc-артефакты с фактическим
  состоянием decision trace, offline eval и KPI monitoring слоя.

## Критерии готовности

- Decision trace API доступен по диалогу, возвращает понятный trace payload и
  покрыт controller/service тестами.
- Offline eval-контур умеет отдавать summary последнего прогона и запускаться
  вручную через AI Ops API без обращения к ручным SQL-операциям.
- AI Ops UI показывает KPI, алерты и offline eval summary на базе backend
  payload без необходимости читать сырые event log записи напрямую.
- Task-артефакты и документация не расходятся с реальным состоянием кода и
  отражают текущий scope задачи.

## Обязательные дополнительные условия

- По завершении задачи показать алерт через
  `./ai-context/baseline/scripts/show-completion-alert.sh`, если пользователь отдельно
  не отменил это требование.
- В финальном ответе подготовить `Резюме для коммита` по формату из
  `ai-context/baseline/guides/tasks.md`.
- После любых file changes создать отдельный changelog-файл в
  `ai-context/changelog` по правилу «одно изменение = один файл» и включить в
  него пользовательский промт, который привел к изменению.
