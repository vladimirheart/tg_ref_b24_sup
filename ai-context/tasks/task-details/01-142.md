# Задача [01-142]

## Короткое описание

Спроектировать чистую целевую архитектуру SQLite-баз: логические имена, финальное число БД и доменные границы.

## Цель

Подготовить понятную и устойчивую target-state архитектуру SQLite-слоя проекта, в которой:

- названия файлов БД отражают их реальную operational-роль;
- количество SQLite-баз соответствует реальным bounded-контекстам проекта, а не историческим наслоениям;
- для каждого домена явно определён один канонический source of truth;
- переход от текущей hybrid/legacy-раскладки к целевой модели можно выполнить поэтапно без скрытых двусмысленностей.

## Контекст

- В `01-141` была зафиксирована текущая operational ownership-модель SQLite и приведён в порядок datasource/bootstrap-слой, но это ещё не означает, что сама topology уже чистая по архитектуре.
- Сейчас primary runtime панели фактически живёт в `tickets.db`, хотя по названию это выглядит как частная база тикетов, а не main operational DB.
- В проекте одновременно присутствуют `tickets.db`, `users.db`, `monitoring.db`, `objects.db`, `settings.db`, `clients.db`, `knowledge_base.db`, `bot_database.db` и per-channel `bot-<channelId>.db`, но не все из них одинаково каноничны по роли.
- Часть split-БД пока выступает как bootstrap/registry или заготовка под будущий bounded-контекст, а часть legacy-данных всё ещё живёт в общей operational БД.
- Нужна отдельная архитектурная задача именно на чистую target topology, чтобы не подменять проектирование постепенными частными переносами.

## Что сделать

- Сформировать целевую карту доменов проекта и определить, какие из них действительно заслуживают отдельную SQLite-БД.
- Определить целевое количество SQLite-баз для проекта и объяснить, почему именно такое разбиение соответствует реальным runtime/use-case границам.
- Предложить логические и устойчивые имена для каждой канонической БД, чтобы имя файла отражало роль БД, а не исторический артефакт.
- Явно разделить:
  - main operational DB панели;
  - auth/users DB;
  - monitoring DB;
  - bot/runtime DB;
  - optional auxiliary/specialized DB, если они действительно нужны.
- Определить, какие текущие БД должны остаться, какие должны быть переименованы, какие должны быть поглощены, а какие — удалены после миграции.
- Подготовить migration plan высокого уровня: порядок переименований/слияний/разделений, требования к backward compatibility, env-переменным, bootstrap и runtime wiring.
- Обновить правила и документацию так, чтобы целевая topology была описана как архитектурное решение, а не как временная заметка в task-detail.

## Критерии готовности

- Для проекта определена одна непротиворечивая целевая topology SQLite-баз.
- Для каждой целевой БД зафиксированы:
  - назначение;
  - канонические домены;
  - причины существования как отдельного файла.
- Имена целевых БД логически соответствуют их реальной роли в проекте.
- Есть явное решение по судьбе `tickets.db`, `settings.db`, `clients.db`, `knowledge_base.db`, `bot_database.db` и per-channel bot DBs.
- Подготовлен поэтапный migration plan, который можно декомпозировать в отдельные implementation-задачи.
