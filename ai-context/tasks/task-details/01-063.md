# Задача [01-063]

Нормализовать legacy `server_type` RMS для chain и kiosk-логики в общем списке.

## Цель

Сделать так, чтобы логика задачи `01-060` работала не только для новых значений `IIKO_CHAIN` / `IIKO_RMS`, но и для уже существующих записей, где тип сервера хранится как `iikoChain` / `iikoOffice`.

## Контекст

- В исходном коде выбор целевой лицензии и kiosk-счётчика уже был привязан к `IIKO_CHAIN` / `IIKO_RMS`.
- В фактических seed-данных и старых RMS-записях в `monitoring.db` используются legacy-значения `iikoChain` и `iikoOffice`.
- Из-за этого общий список RMS продолжал показывать fastfood-логику даже там, где ожидались `FullEdition (Server)` и kiosk-бейдж.

## Что сделать

- В `RmsLicenseMonitoringService` добавить нормализацию `server_type` в единый ключ.
- В DTO RMS вернуть нормализованные `server_type_key` и `server_type_display`.
- В UI RMS использовать нормализованный ключ для подсветки chain-строк и отображения kiosk-бейджа.
- В seed-импорте сразу сохранять нормализованный тип сервера для новых записей.

## Критерии готовности

- Для записей с `server_type=iikoChain` общий список RMS использует chain-логику `FullEdition (Server) / id=10`.
- Для записей с `server_type=iikoOffice` общий список RMS использует RMS-логику и показывает kiosk-бейдж `id=36073118`.
- Подсветка строк chain и тип сервера в UI больше не зависят от сырого legacy-значения.
