# Задача [01-077]

## Короткое описание

Показать результат sync департаментов и сохранять ушедшие локации как закрытые.

## Цель

Сделать результат синхронизации наблюдаемым и безопасным для исторических данных: оператор
должен понимать, что поменялось после sync, а пропавшие из `iikoServer` локации не должны
исчезать бесследно — их нужно переводить в закрытое состояние.

## Контекст

- В location sync уже есть progress/status контур, но сам бизнес-результат sync должен быть
  прозрачен пользователю.
- Effective location catalog используется не только на странице настроек, но и в смежных flows,
  поэтому физическое исчезновение локации может ломать ссылки, историю и downstream-поведение.
- Для `Track A3` это финальный baseline-пакет, который закрывает вопрос не только получения
  данных, но и безопасного жизненного цикла записей.
- Задача особенно важна перед последующим module split, потому что после разрезания settings runtime
  скрытые lifecycle-решения будут ещё труднее отлавливать.

## Пересечения с backlog

- `01-074` даёт оператору ручной sync и progress, а `01-077` добавляет осмысленный итог sync.
- `01-073` нужен для безопасного мёрджа при сложной структуре данных.
- `01-092` и другие consumers effective locations выигрывают от того, что ушедшие записи не
  удаляются жёстко и могут быть корректно интерпретированы как закрытые.
- `01-128` / `01-129` должны сохранить lifecycle semantics закрытых локаций при split settings runtime.

## Что сделать

- Показывать оператору итог sync: сколько локаций добавлено, обновлено, закрыто, какие есть предупреждения.
- Переводить локации, исчезнувшие из live-source, в закрытое состояние вместо физического удаления.
- Сохранить совместимость effective payload и downstream consumers с новым состоянием закрытых локаций.
- Зафиксировать поведение тестами на merge/result lifecycle.

## Критерии готовности

- После sync пользователь видит понятный summary результата.
- Ушедшие из `iikoServer` локации не удаляются бесследно, а сохраняются как закрытые.
- Смежные consumers корректно работают с закрытыми локациями и не получают broken payload.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из
  `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog` и включить в него
  пользовательский промпт, который привёл к изменению.
