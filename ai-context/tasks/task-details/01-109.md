# Задача [01-109]

Починить два связанных Telegram-сценария:

- после выбора ресторана/локации бот не должен задавать случайный тестовый вопрос `укывеапрр`, а должен переходить к штатному шагу описания проблемы;
- ответ оператора из панели должен отправляться через тот же Telegram Bot API base URL и сетевой маршрут канала, без минутного подвисания кнопки `Отправить`.

## Контекст

- в `config/shared/settings.json` в активном `question_flow` и шаблоне остался тестовый custom-вопрос после `location_name`;
- `DialogReplyTransportService` продолжал ходить напрямую в `https://api.telegram.org`, игнорируя уже добавленные base URL / mirror / proxy-настройки канала;
- из-за этого операторский reply зависал до системного таймаута Java HTTP client и визуально блокировал кнопку отправки.

## Что сделать

- удалить лишний custom-вопрос из bot settings question flow;
- перевести Telegram send/edit/delete/media в `DialogReplyTransportService` на `IntegrationNetworkService` и channel-aware Bot API URL;
- добавить явный request timeout для Telegram reply transport;
- зафиксировать изменения в append-only changelog.

## Критерии готовности

- после `location_name` бот спрашивает `Опишите проблему`, если в сценарии нет других осознанно настроенных шагов;
- reply/edit/delete/media из панели используют channel `base_url` / legacy mirror и сетевой маршрут;
- зависание кнопки `Отправить` больше не упирается в минутный direct timeout до `api.telegram.org`;
- `spring-panel` компилируется.
