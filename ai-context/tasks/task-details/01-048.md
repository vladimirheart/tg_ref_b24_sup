# Задача [01-048]

## Короткое описание

Починить 500 на странице RMS-мониторинга при пустом состоянии очередей.

## Цель

Сделать так, чтобы `/analytics/rms-control` корректно открывалась до первого
ручного или автоматического запуска очередей обновления.

## Контекст

API `GET /api/monitoring/rms/sites` возвращал `500 Internal Server Error`, хотя
список RMS-записей в базе уже есть. Причина оказалась в
`RmsLicenseMonitoringApiController`: метод `toRefreshState(...)` использовал
`Map.of(...)`, а `Map.of` не допускает `null`. На первом открытии страницы поля
`last_requested_at` и `last_completed_at` у очередей ещё `null`, из-за чего
контроллер падал до отдачи списка.

## Что сделать

- Убрать использование `Map.of(...)` для `refresh_state`, где допустимы `null`.
- Вернуть сериализацию через обычные `LinkedHashMap`.
- Проверить сборку `spring-panel`.

## Критерии готовности

- `GET /api/monitoring/rms/sites` не падает на начальном состоянии очередей.
- Страница `/analytics/rms-control` может получить список RMS до первого refresh.
- `spring-panel\\mvnw.cmd -q -DskipTests compile` проходит успешно.

## Обязательные дополнительные условия

  `ai-context/baseline/guides/tasks.md`.

  `ai-context/changelog` по правилу «одно изменение = один файл».
