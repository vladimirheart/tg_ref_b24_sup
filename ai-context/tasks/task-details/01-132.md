# Задача [01-132]

## Короткое описание

Разрезать AnalyticsController по bounded governance и analytics use-cases.

## Цель

Убрать из `AnalyticsController` смешение page rendering, export, workspace
rollout/context endpoints, SLA governance и macro governance mutation flows.

## Контекст

- По audit-срезу `AnalyticsController` держит около `1124` строк.
- Внутри него собраны page render routes, export, workspace rollout review,
  workspace context standardization, legacy usage policy, SLA governance review
  и macro governance review/policies.
- Это secondary controller hotspot из `Track C`, который логично брать после
  UI runtime split или параллельно узкими пакетами.
- Вокруг analytics/dashboard уже есть отдельный пользовательский backlog
  `01-118`..`01-127`, поэтому controller split должен сохранить текущие
  dashboard и analytics contracts, а не смешать transport refactor с новым UI
  редизайном.

## Пересечения с backlog

- `01-118`..`01-127` — dashboard payload и analytics page flow считаются
  действующим UI baseline.
- `01-024` — macro/SLA/workspace governance already moved toward thin services,
  поэтому здесь нужен transport split, а не повторный service-level refactor.

## Что сделать

- Разделить `AnalyticsController` минимум на `analytics page/export`,
  `workspace rollout governance`, `workspace context`, `SLA governance`,
  `macro governance`.
- Оставить controller-слой transport-oriented, не перенося в него новую
  orchestration-логику.
- Синхронизировать WebMvc coverage и связанные tests под новую раскладку.
- Сохранить текущие endpoint contracts для UI и внутренних governance flows.

## Критерии готовности

- `AnalyticsController` больше не является единственной точкой концентрации
  rollout/context/SLA/macro governance transport-слоя.
- Новые controllers разделены по bounded use-cases и остаются thin wrappers.
- Существующий analytics/governance UI продолжает работать без смены внешнего
  API-контракта.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в `ai-context/changelog` и включить в него пользовательский промт, который привёл к изменению.
