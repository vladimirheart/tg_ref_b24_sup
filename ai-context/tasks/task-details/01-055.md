# Задача [01-055]

## Короткое описание

Снизить блокировки SQLite в `spring-panel` и добить восстановление имени RMS из битой строки вида
`╨д╨а...`.

## Цель

Убрать частые runtime-падения из-за `SQLITE_BUSY`/`SQLITE_BUSY_SNAPSHOT` на `SPRING_SESSION`,
`rms_license_monitors` и `ssl_certificate_monitors`, а также добиться, чтобы столбец `Имя` на
странице контроля RMS не затирался битым значением после сохранения пароля и очередных refresh.

## Контекст

- По логу `C:\Users\SinicinVV\git_h\tg_ref_b24_sup\logs\spring-panel.log` массовые ошибки
  начинаются с `UPDATE SPRING_SESSION`, а затем цепляют `/error`, `/api/channels`,
  `/api/unblock-requests/summary` и ручной refresh SSL/RMS.
- В `SslCertificateMonitoringService` длинная транзакция может жить во время сетевого SSL-запроса,
  что опасно для SQLite и объясняет `SQLITE_BUSY_SNAPSHOT`.
- Текущий хелпер RMS лечит один тип mojibake, но пользовательский пример `╨д╨а...` указывает на
  отдельный путь перекодировки, который нужно распознавать явно.

## Что сделать

- Усилить конфигурацию SQLite для primary и monitoring datasource.
- Убрать длинную транзакцию вокруг сетевого SSL-check и добавить мягкий retry для monitoring writes.
- Расширить восстановление RMS-имени под строку mojibake с символами `╨`/`╤`.
- Зафиксировать выводы по `spring-panel.log` в ответе пользователю.

## Критерии готовности

- `mvnw -q -DskipTests compile` проходит.
- В коде больше нет длинной monitoring-транзакции, охватывающей SSL socket check.
- Monitoring-репозитории умеют переживать краткие `SQLITE_BUSY` без немедленного падения.
- RMS `serverName` восстанавливается и не должен сохраняться в виде `╨д╨а...`.

  `ai-context/baseline/guides/tasks.md`.
