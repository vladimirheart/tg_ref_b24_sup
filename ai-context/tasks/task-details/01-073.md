# Задача [01-073]

Починить сохранение `Структуры локаций`, которое падает на `SQLITE_CONSTRAINT_UNIQUE` при синхронизации справочных параметров в `settings_parameters`.

## Цель

Сделать сохранение локаций устойчивым, даже если одно и то же значение `city` или другого параметра встречается в нескольких ветках дерева локаций.

## Контекст

- Пользователь получил ошибку при сохранении:
  `UNIQUE constraint failed: settings_parameters.param_type, settings_parameters.value`.
- В коде `SettingsParameterService` синхронизация из location tree пыталась вставлять несколько записей с одинаковой парой `(param_type, value)`, если различались зависимости.
- Реальная схема БД запрещает такие дубли уникальным индексом только по `(param_type, value)`.

## Что сделать

- Выровнять `SettingsParameterService` под фактическую схему таблицы.
- Не вставлять повторные записи при одинаковых `(param_type, value)`.
- При наличии существующей записи аккуратно дополнять `extra_json`, если в нём не хватает зависимостей.
- Добавить тест на сценарий с одинаковым городом в нескольких чейнах.

## Критерии готовности

- Сохранение `Структуры локаций` не падает на `UNIQUE constraint failed`.
- Синхронизация параметров из location tree не создаёт дублей по `(param_type, value)`.
- Тесты покрывают реальную уникальность таблицы `settings_parameters`.
