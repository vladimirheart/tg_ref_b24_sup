# Задача [01-083]

Восстановить запуск `MAX`-бота после регрессии и убрать ложноположительный статус старта, когда процесс успевает вывести `Started ...`, но падает сразу после этого.

## Цель

- вернуть рабочий автозапуск `MAX` через `spring-panel`;
- передавать путь к общей SQLite-базе боту через прямой Spring runtime-contract, а не только через косвенный env fallback;
- сделать подтверждение готовности процесса устойчивым к раннему падению после startup marker;
- сохранить честную диагностику для `Telegram`, который сейчас всё ещё блокируется unsupported `mtproto`.

## Контекст

- После изменений в dialogs UI пользователь сообщил, что не запускаются оба бота: `MAX` и `Telegram`.
- Разбор логов показал две независимые причины:
  - `Telegram` падает на неподдерживаемой runtime-схеме прокси `mtproto`.
  - `MAX` доходил до старта web/runtime, но падал на `SqliteTriggerInitializer` с `no such table: main.tickets`.
- Дополнительная проверка показала, что таблица `tickets` есть в общей panel-базе, а проблема была в ненадёжной передаче DB path в bot runtime.
- Панель также считала бот запущенным слишком рано: сразу после строки `Started ...`, без короткого окна стабилизации.

## Что сделать

- В `BotRuntimeContractService` добавить прямые runtime env:
  - `SUPPORT_BOT_DATABASE_PATH`
  - `SPRING_SQL_INIT_MODE=always`
- Сохранить совместимость с существующим `APP_DB_TICKETS`.
- В `BotProcessService` добавить окно стабилизации после startup marker и не считать процесс готовым, если он умер сразу после `Started ...`.
- Обновить/добавить тесты на runtime-contract и readiness-check.

## Критерии готовности

- `MAX`-бот стартует через `spring-panel` и остаётся живым после автозапуска.
- В `support-bot-max-2-process.log` больше нет падения `no such table: main.tickets`.
- `spring-panel` не помечает бот как running, если тот упал сразу после startup marker.
- Целевые тесты по runtime-contract и process-readiness проходят.

## Дополнительные условия

- `Telegram` в этой задаче не переводится в working-state, пока для него не будет задан поддерживаемый proxy/network route вместо `mtproto`.
- После file changes создать отдельный changelog в `ai-context/changelog`.
- В финальном ответе подготовить `Резюме для коммита`.
