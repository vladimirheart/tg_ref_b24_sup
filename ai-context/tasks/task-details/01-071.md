# Задача [01-071]

## Короткое описание

Перевести источник департаментов локаций с `iiko Transport` на `iikoServer API`.

## Цель

Сделать `iikoServer API` единственным live-источником департаментов для блока `Структура локаций`,
чтобы дерево локаций и его downstream-потребители больше не зависели от transport API и
активного `iiko biz`-профиля.

## Контекст

- В текущем task-контуре `01-071` является первым baseline-пакетом для `locations/iikoServer`
  ветки, на которую уже ссылаются `01-024`, `01-128` и `01-129`.
- На странице настроек уже есть отдельная вкладка `Синхронизация iikoServer`, а effective payload
  для location-структуры собирается через
  `IikoDepartmentLocationCatalogService.loadCatalog()` и
  `buildEffectiveLocationsPayload(...)`.
- На смежных экранах уже зафиксировано, что transport API и `iikoServer API` — разные контуры:
  например, `analytics/iiko-api-monitoring.html` прямо отделяет transport monitoring от
  синхронизации структуры локаций.
- Эту задачу нельзя реализовывать как новый UI-redesign: она должна зафиксировать именно source-of-truth
  для `locations` runtime перед более поздним split `settings/index.html`.

## Пересечения с backlog

- `01-066` — live-структура локаций уже была выделена как отдельный проблемный контур;
  `01-071` уточняет, что live-source для департаментов должен быть именно `iikoServer API`.
- `01-072`..`01-077` — последующие задачи этого же пакета опираются на то, что источник уже
  переведён и больше не смешивается с transport-сценариями.
- `01-092` — страница паспорта объекта должна читать те же effective locations, что и настройки,
  поэтому source switch нельзя делать только “для одной страницы”.
- `01-128` / `01-129` — будущий settings runtime split обязан сохранить уже введённую source boundary.

## Что сделать

- Перевести загрузку live-департаментов для location-каталога на `iikoServer API`.
- Убрать зависимость location sync/runtime от `iiko Transport` API там, где речь идёт именно
  о live-структуре департаментов.
- Сохранить текущий effective payload contract для `settings`, `management/passports`,
  `public form` и других consumers, чтобы source switch не ломал клиентский слой.
- Явно зафиксировать, что transport API продолжает жить только в своём bounded-сценарии, без
  повторного смешения с location-source логикой.

## Критерии готовности

- Департаменты в `Структуре локаций` и связанных payload строятся из `iikoServer API`, а не из
  transport API.
- Смежные consumers (`settings`, `passports`, `public form`) получают тот же effective contract,
  но уже с новым source-of-truth.
- В коде и UI больше нет двусмысленности, что location departments могут подтягиваться из transport API.
