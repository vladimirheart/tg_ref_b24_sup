# Задача [01-071]

Перевести live-источник департаментов для структуры локаций с iiko Transport API на iikoServer API через `GET /resto/api/corporation/departments/`, не ломая существующий мониторинг transport-запросов.

## Цель

Сделать так, чтобы каскадные поля локаций и блок `Структура локаций` строились по данным `iikoServer API`, а не по transport-monitoring записям `organizations`.

## Контекст

- Текущая реализация `IikoDepartmentLocationCatalogService` использует `iiko_api_monitors` и transport endpoints `/api/1/access_token` + `/api/1/organizations`.
- Пользователь уточнил, что transport-monitoring должен остаться как отдельный мониторинговый контур.
- Для департаментов нужно использовать documented endpoint `https://host:port/resto/api/corporation/departments/` из `iikoServer API`.
- В проекте уже есть существующий `iikoServer` credential source в `EmployeeDiscountAutomationCredentialService` с `baseUrl`, `apiLogin`, `apiSecret`.

## Что сделать

- Отвязать `IikoDepartmentLocationCatalogService` от transport monitor-источников.
- Подключить получение credentials из существующего `iikoServer`-профиля.
- Реализовать HTTP gateway под `iikoServer API` для авторизации и загрузки департаментов.
- Сохранять текущую бизнес-логику парсинга: только активные департаменты, исключать `CLOSED`, разбирать `ФР_`, `ББ`, `СВ`, город и локацию.
- Обновить тесты и связанные места, чтобы live-локации больше не зависели от transport monitors.

## Критерии готовности

- `Структура локаций` и клиентские вопросы строятся по `iikoServer API`.
- Transport monitoring продолжает работать без изменения своего поведения.
- Если `iikoServer`-профиль не настроен или ответ пустой, остаётся fallback на shared config.
- Новая реализация покрыта точечными тестами.
