# Задача [01-058]

Унифицировать UI мониторингов: модалки добавления, overview доступности и сортировку RMS.

## Цель

Сделать страницы мониторинга SSL, RMS и iiko API визуально и поведенчески согласованными: добавление новых записей через отдельные модалки, обзор доступности на верхнем уровне и удобная работа с RMS-списком через фильтры, компактные индикаторы и сортировку.

## Контекст

- В проекте уже есть отдельный monitoring-контур и страница `/analytics/rms-control` с RMS overview.
- SSL-мониторинг уже поддерживает клиентскую сортировку, но ещё использует inline-форму добавления и не показывает общий overview доступности.
- iiko API-мониторинг тоже пока использует inline-форму добавления.
- Нужно соблюдать правила:
  - `C:\Users\SinicinVV\git_h\tg_ref_b24_sup\ai-context\baseline\guides\tasks.md`
  - `C:\Users\SinicinVV\git_h\tg_ref_b24_sup\ai-context\rules\backend\03-monitoring-overview-availability.md`

## Что сделать

- Перевести добавление новых записей на страницах SSL и iiko API в отдельные модальные окна по аналогии с RMS.
- Добавить на страницу SSL верхнеуровневую плашку общей доступности и фильтрацию списка по доступности.
- Доработать RMS-страницу:
  - фильтр по статусам из overview;
  - сортировка по столбцам;
  - убрать логин из общего списка;
  - показывать дату окончания лицензии без времени;
  - вернуть количество целевой лицензии `RMS (Front Fast Food)` в общем списке;
  - заменить текстовые индикаторы enable/disable на компактные иконки;
  - подсвечивать строки `IIKO_CHAIN`.

## Критерии готовности

- SSL и iiko API создают записи через отдельные модалки, а не через inline-формы.
- SSL и RMS показывают overview общей доступности, а RMS-список фильтруется по статусам из overview.
- RMS-таблица сортируется по заголовкам и отображает более компактный список без логина и без времени в expiry.
