# Задача [01-134]

## Короткое описание

Выпилить оставшийся backend/test слой `PublicForm` после удаления публичных
форм из проекта.

## Цель

- убрать из живого кода и тестового контура остаточный `PublicForm`-subsystem,
  который больше не соответствует текущему product scope;
- зачистить связанные watcher/audit-ветки и integration/template tests,
  завязанные на `public_form_*`;
- зафиксировать removal как отдельный пакет, а не смешивать его с cleanup
  settings runtime из `01-129`.

## Контекст

- в рамках `01-129` из `settings/index.html` уже удалены JS-хвосты
  `public_form`; это был только локальный cleanup settings-страницы, а не
  удаление всего subsystem;
- пользователь отдельно подтвердил, что публичные формы выпилены из проекта и
  не интересуют текущий продуктовый этап;
- по состоянию на `2026-06-25` в репозитории ещё остаются backend/test
  references на `PublicForm`:
  `spring-panel/src/main/java/com/example/panel/service/OperatorNotificationWatcher.java`,
  `spring-panel/src/test/java/com/example/panel/service/OperatorNotificationWatcherTest.java`,
  `spring-panel/src/test/java/com/example/panel/service/SupportPanelIntegrationTests.java`,
  `spring-panel/src/test/java/com/example/panel/controller/PublicShellTemplateContractTest.java`;
- отдельные исторические задачи и документы уже касались этого контура:
  `01-086`, `01-129`,
  `docs/ARCH_UI_REFACTORING_ROADMAP_2026-04-15.md`,
  `docs/ARCHITECTURE_AUDIT_2026-04-08.md`.

## Что сделать

- собрать полный актуальный inventory оставшихся `PublicForm`-классов,
  моделей, контроллеров, сервисов, шаблонов, audit actions и tests вне
  settings cleanup;
- удалить или заменить backend/runtime-ветки, которые ещё ожидают
  `public_form_submit`, `public_form_new_appeal_notification` и соседний
  `PublicForm` flow;
- подчистить integration/unit/template tests и документацию, чтобы в проекте не
  оставалось ложного active-scope вокруг удалённой функциональности;
- отдельно проверить, не висят ли на этом контуре миграции, config keys,
  маршруты, шаблоны, security rules и runtime notifications.

## Критерии готовности

- в рабочем backend/runtime коде больше нет активного `PublicForm`-flow,
  который может сработать в production;
- тестовый контур больше не держит сценарии, основанные на удалённых публичных
  формах, либо эти сценарии явно переписаны под новый продуктовый scope;
- `rg` по `PublicForm|public_form|publicForm|PublicShell` не показывает
  несанкционированных хвостов вне допустимых исторических записей
  `ai-context/changelog`, backlog и audit-документов;
- removal не ломает сборку и соседние сценарии operator notifications /
  dialog history.

## Обязательные дополнительные условия

- В финальном ответе подготовить `Резюме для коммита` по формату из
  `ai-context/baseline/guides/tasks.md`.
- После file changes создать отдельный changelog-файл в
  `ai-context/changelog` и включить в него пользовательский промт, который
  привёл к изменению.
