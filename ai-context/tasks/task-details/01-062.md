# Задача [01-062]

Подтянуть активные департаменты из iiko в каскадные вопросы клиентской формы.

## Цель

Сделать источник локаций для клиентских вопросов живым: получать активные департаменты из iikoServer API, исключать закрытые точки и возвращать в публичную форму корректный каскад `бизнес -> тип -> город -> локация`.

## Контекст

- Пользователь указал `https://ru.iiko.help/articles/#!api-documentations/iikoserver-api` как источник данных и описал правила разбора названий департаментов.
- В проекте уже есть иерархия `business -> location_type -> city -> location_name` в `config/shared/locations.json`, но сейчас она статична.
- В `spring-panel` уже есть runtime для iiko API monitoring с поддержкой `/api/1/access_token`, `/api/1/organizations` и `/api/1/terminal_groups`.
- Публичная форма сейчас умеет только простые статические `select`, без зависимых списков.
- В `config/shared/settings.json` и в боте уже используются поля `business`, `location_type`, `city`, `location_name`, поэтому новую логику лучше оформить как переиспользуемый каталог, а не как одноразовый хардкод только для формы.

## Что сделать

- Добавить сервис, который получает активные `terminal_groups` из iiko API, исключает элементы с `CLOSED` в имени и раскладывает точки в дерево `business -> type -> city -> location`.
- Использовать правила разбора названия департамента:
  - `ФР_` означает франчайзи;
  - `ББ` и `ФР_ББ` относятся к chain `БлинБери`;
  - `СВ` и `ФР_СВ` относятся к chain `СушиВёсла`;
  - город идёт сразу после префикса chain;
  - остаток строки после города считается названием локации.
- В качестве справочника business использовать chain-слой из RMS monitoring, а при недоступности внешнего каталога падать обратно на текущий `locations.json`.
- Расширить пресеты локаций до зависимых списков с `tree` и `option_dependencies`, чтобы их можно было использовать и в боте, и в публичной форме.
- Подключить новый каталог к `PublicFormService`/`PublicFormApiController`, чтобы поля `business`, `location_type`, `city`, `location_name` отдавались как каскадные `select`.
- Обновить `public-form.js`, чтобы при смене предыдущего поля пересчитывались доступные варианты и сбрасывались невалидные значения.
- Добавить серверную валидацию комбинаций полей, чтобы нельзя было отправить несуществующую связку бизнеса, типа, города и локации.

## Критерии готовности

- Публичная форма получает для location-полей только активные департаменты без `CLOSED`.
- Для `ФР_*` точек тип определяется как франчайзи, для остальных как корпоративная сеть.
- Бизнес, тип, город и локация в форме работают каскадно и не позволяют выбрать несовместимые значения.
- При недоступности iiko API форма продолжает работать на fallback-данных из `locations.json`.
- Тесты покрывают хотя бы построение каталога и API-ответ публичной формы с каскадными полями.
