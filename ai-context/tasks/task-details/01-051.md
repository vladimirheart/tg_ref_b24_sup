# Задача [01-051]

## Короткое описание

Добавить в аналитику мониторинг выбранных запросов iiko API с разными `apikey`.

## Цель

Дать возможность заводить несколько независимых проверок iiko API с разными
ключами, выбирать тип запроса из безопасного read-only набора и видеть текущее
состояние каждой проверки в отдельной странице аналитики.

## Контекст

Пользователь попросил сначала изучить официальную документацию
`https://api-ru.iiko.services/docs`, а затем встроить мониторинг выбранных
запросов. В официальном OpenAPI `iikoCloud API` есть удобная цепочка для
мониторинга: `/api/1/access_token` возвращает bearer token по `apiLogin`, после
чего можно вызывать read-only методы вроде `/api/1/organizations`,
`/api/1/terminal_groups`, `/api/1/nomenclature`, `/api/1/stop_lists` и
`/api/2/menu/by_id`. В проекте уже есть отдельная `monitoring.db` и готовые
паттерны для аналитических мониторингов SSL и RMS, их нужно переиспользовать,
а не создавать новый разрозненный контур.

## Что сделать

- Добавить в `monitoring.db` новую таблицу для iiko API monitors.
- Реализовать backend-сервис, который получает `access_token`, вызывает
  выбранный метод iiko API и сохраняет статус, HTTP-код, длительность, ошибку и
  краткую сводку ответа.
- Добавить scheduler и очередь ручных/автоматических запусков для iiko API
  мониторинга.
- Добавить новую страницу в разделе аналитики с таблицей, CRUD, массовым
  включением/отключением и ручным запуском проверок.
- Поддержать разные `apikey` на уровне отдельных записей и параметры,
  зависящие от выбранного метода.

## Критерии готовности

- В аналитике есть отдельная страница мониторинга iiko API.
- Можно создать несколько записей с разными `apikey` и разными типами запросов.
- Мониторинг умеет автоматически и вручную запускать проверки через очередь.
- В UI видны статус, код ответа, длительность, ошибка и краткая сводка
  последнего ответа.
- `spring-panel\\mvnw.cmd -q -DskipTests compile` проходит успешно.
