# Задача [01-066]

## Короткое описание

Перевести live-структуру локаций `iiko` на `organizations` и выделить source-записи мониторинга.

## Цель

Сделать `organizations` канонической моделью live-source для структуры локаций и явно отделить
monitoring source records от остальных iiko API checks, чтобы location-синхронизация строилась
на понятной и устойчивой upstream-базе.

## Контекст

- В `IikoApiMonitoringService` уже есть явное правило: source структуры локаций можно включать
  только для request type `organizations`, а в UI monitoring это отделено от остальных request types.
- Позднейший пакет `01-071`..`01-077` уже развивает тему `locations/iikoServer`, но upstream-решение
  про `organizations` и source-record boundary должно быть зафиксировано отдельно.
- В project docs и task-контуре уже видно, что `locations.json`, `org_structure.json` и
  effective locations catalog используются несколькими consumers; значит source-запись мониторинга
  должна быть явной, а не смешанной с произвольными read-only iiko checks.
- Эта задача не про общий мониторинговый UI-redesign, а про архитектурную границу между
  location-source monitoring и остальными integration diagnostics.

## Пересечения с backlog

- `01-062` потребляет результат в клиентской форме и зависит от корректной upstream live-структуры.
- `01-071` уточняет следующий шаг: после перевода live-source на `organizations` нужно ещё
  перевести сами департаменты локаций на `iikoServer API`.
- `01-072`..`01-077` продолжают этот bounded-контур уже на settings/runtime уровне.
- `01-118` использует `locations.json` как источник для dashboard-агрегаций, поэтому upstream
  location structure не должна оставаться двусмысленной.

## Что сделать

- Зафиксировать `organizations` как единственный допустимый request type для live-source структуры локаций.
- Выделить monitoring source records, которые реально участвуют в location sync, из общего набора
  iiko API monitor entries.
- Развести в UX и в backend семантику “это monitor для диагностики” и “это source для live-структуры локаций”.
- Сохранить совместимость со downstream consumers effective locations catalog.

## Критерии готовности

- Для source структуры локаций используется именно `organizations`, а не произвольные iiko request types.
- Source-записи мониторинга различимы и не смешиваются с обычными диагностическими monitors.
- Downstream location consumers получают более ясный и устойчивый upstream contract.
