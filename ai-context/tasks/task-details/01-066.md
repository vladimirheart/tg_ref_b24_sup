# Задача [01-066]

Перевести live-структуру локаций iiko на `organizations` и выделить source-записи мониторинга.

## Цель

Сделать так, чтобы live-структура локаций для публичной формы и настроек строилась по названиям организаций iiko, а не по `terminal_groups`, и чтобы в мониторинге iiko API можно было явно отметить записи, которые используются как источник этого справочника.

## Контекст

- Сейчас `IikoDepartmentLocationCatalogService` получает список `organizationIds`, а затем тянет названия из `/api/1/terminal_groups`.
- Пользователь уточнил, что для структуры локаций нужно использовать именно `organizations`, потому что naming convention живёт в названиях организаций.
- На странице `Мониторинг iiko API` могут храниться разные типы запросов, поэтому нельзя брать все enabled-записи подряд: нужен отдельный флаг источника локаций.
- Monitoring-данные живут в `monitoring.db` и схема этой БД поддерживается через `MonitoringDatabaseBootstrapService`, а не через обычные Flyway-миграции primary DB.

## Что сделать

- Добавить в модель/хранилище iiko monitor явный boolean-флаг использования записи для синхронизации структуры локаций.
- Обновить backend мониторинга и UI страницы `Мониторинг iiko API`, чтобы флаг можно было видеть и редактировать только для релевантных записей.
- Перевести `IikoDepartmentLocationCatalogService` на получение и парсинг названий организаций из `/api/1/organizations`, используя только отмеченные source-записи.
- Обновить тесты для новой логики source-записей и построения live-структуры.

## Критерии готовности

- В live-структуру локаций попадают названия организаций из iiko `organizations`, а не `terminal_groups`.
- Для синхронизации локаций используются только monitor-записи с явным флагом источника локаций.
- Страница мониторинга позволяет увидеть и сохранить этот флаг, а сервер не принимает несовместимую комбинацию флага и request type.
