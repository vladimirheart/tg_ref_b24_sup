# Задача [01-151]

Починить запуск `spring-panel` после конфликта Flyway вокруг legacy-миграции `client_phones`: старая версия `6.1` стала запоздалой, а предыдущий автозапуск через `repair()` добавил ложные `DELETE`-записи в `flyway_schema_history`.

## Цель

- восстановить штатный запуск `spring-panel` на существующей `panel_runtime.db`;
- убрать риск повторной порчи `flyway_schema_history` через общий `repair()`;
- сохранить корректную, идемпотентную миграцию схемы `client_phones`.

## Контекст

- `spring-panel/run-windows.bat` 22 июля 2026 года падал на `FlywayValidateException`: локальная история базы не принимала late migration `6.1`;
- старая Java-миграция `V6_1__fix_client_phones_schema` лежала вне `db.migration.sqlite`, тогда как приложение запускает SQLite Flyway с `classpath:db/migration/sqlite`;
- предыдущий старт уже успел дописать в `panel_runtime.db` строки `type='DELETE'` для исторических версий `1..34`, из-за чего Flyway пытался переигрывать базовые SQL-миграции и падал на дублях колонок;
- runtime SQLite-контур для панели закреплён за `panel_runtime.db` и описан в проектном правиле топологии SQLite.

## Что сделать

- перенести правку схемы `client_phones` в новую хвостовую SQLite Java-миграцию с безопасной версией;
- удалить legacy-класс `V6_1__fix_client_phones_schema`, который больше не должен участвовать в истории;
- заменить общий `flyway.repair()` на точечную нормализацию `flyway_schema_history` перед `migrate()`;
- проверить `clean compile` и реальный запуск `run-windows.bat` на текущей локальной базе.

## Критерии готовности

- `spring-panel` больше не падает на `resolved migration not applied` для `6.1`;
- `spring-panel` не пытается заново выполнять `V1..V34` из-за ложных `DELETE`-маркеров;
- лог старта подтверждает нормализацию истории, применение новой миграции и успешный запуск приложения.

- в финальном ответе подготовить `Резюме для коммита`.
