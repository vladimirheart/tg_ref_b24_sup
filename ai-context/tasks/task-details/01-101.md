# Задача [01-101]

Починить preview медиа в legacy-модалке диалога: операторские вложения должны открываться по рабочему attachment URL, а `Esc` внутри просмотрщика изображения должен закрывать именно просмотрщик, не родительскую модалку диалога.

## Цель

- вернуть отображение картинок и другого media, отправленного оператором из диалога;
- сохранить совместимость с уже существующими attachment-путями в истории;
- сделать поведение `Esc` предсказуемым для вложенного media preview.

## Контекст

- `DialogConversationReadService` нормализует поле `chat_history.attachment` в URL, который потом рендерится в `dialogs.js`;
- операторские media reply сохраняют в `chat_history.attachment` только `storedName`, а публичный URL для них обслуживается через `AttachmentController` на `/api/attachments/tickets/{ticketId}/{filename}`;
- клиентские вложения исторически могут храниться как файловые пути внутри `attachments/...`, поэтому для них уже существует endpoint `/api/attachments/tickets/by-path`;
- `dialogs.js` открывает media preview отдельной Bootstrap modal поверх `dialogDetailsModal`, но без явного перехвата `Escape`.

## Что сделать

- исправить генерацию attachment URL в `DialogConversationReadService` для операторских `storedName`;
- добавить обратную совместимость для path-based вложений через `/api/attachments/tickets/by-path`;
- добавить/обновить тесты на построение attachment URL;
- перехватить `Escape` для `dialogMediaPreviewModal` раньше родительской dialog modal.

## Критерии готовности

- media, отправленное оператором, снова отображается в истории диалога как картинка/видео, а не падает в `500`;
- path-based клиентские вложения продолжают открываться через поддерживаемый endpoint;
- при открытом preview изображения клавиша `Esc` закрывает preview и не закрывает `dialogDetailsModal`.
