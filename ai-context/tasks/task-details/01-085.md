# Задача [01-085]

Сделать удаление на странице `analytics/rms-control` устойчивым к перезапуску проекта: запись не должна физически удаляться из базы, но после удаления обязана исчезать из списка и не возвращаться обратно seed-импортом.

## Цель

- сохранить soft-delete поведение для RMS-мониторов;
- перестать показывать скрытые записи в `rms-control`;
- не допускать повторного появления скрытых seed-записей после рестарта;
- позволить повторно добавить ранее скрытую RMS тем же адресом без конфликта по unique-индексу.

## Контекст

- сейчас API удаления RMS удаляет запись физически;
- на старте `RmsMonitoringSeedImportService` снова импортирует адрес из seed CSV, если не находит его в таблице;
- из-за этого удалённые seed-RMS после перезапуска снова появляются в интерфейсе;
- полное удаление для этой сущности нежелательно, нужен режим скрытия из списка.

## Что сделать

- добавить в `rms_license_monitors` поля soft-delete (`is_deleted`, `deleted_at`) с bootstrap-миграцией для существующей `monitoring.db`;
- перевести репозиторий и сервис RMS-мониторинга на soft-delete вместо `DELETE`;
- исключить скрытые записи из обычной выдачи списка и операций по видимым RMS;
- учитывать скрытые записи как уже известные для seed-импорта;
- при повторном добавлении RMS с тем же адресом восстанавливать скрытую запись, а не падать на unique-ограничении.

## Критерии готовности

- после удаления RMS пропадает из списка `rms-control`;
- после перезапуска проекта скрытая seed-RMS не возвращается в список;
- запись остаётся в `monitoring.db` как скрытая;
- повторное добавление того же `rms_address` восстанавливает запись в мониторинге.
