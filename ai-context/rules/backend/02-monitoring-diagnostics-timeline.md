# Правило: Диагностика мониторингов должна показывать историю состояний

## Статус

Действует.

## Правило

Если в `spring-panel` для мониторинга есть отдельная диагностическая страница, модалка или detail-view, она не должна ограничиваться только текущим или последним состоянием проверки.

Для таких представлений нужно выводить временную шкалу последних событий проверки:

- время события;
- тип проверки или канал проверки;
- итоговый статус;
- краткую сводку результата;
- при наличии технические поля уровня `HTTP`, длительности, route summary и similar low-level context.

## Область применения

- RMS diagnostics;
- iiko API diagnostics / last response views;
- будущие monitoring views для SSL, HTTP, DNS, SMTP и других health-check подсистем.

## Практический смысл

- оператор видит не только "что сейчас", но и "как это менялось во времени";
- легче отличать разовый сбой от деградации или флапа;
- проще разбирать проблемы доступности без прямого доступа к runtime-логам и БД;
- новые мониторинги получают единый UX-стандарт диагностики.
