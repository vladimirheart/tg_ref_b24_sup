# Tasks Guide

Этот документ описывает только baseline-правила task-flow.
Живые данные задач всегда лежат в `ai-context/tasks/`.

## Где что находится

- `tasks/task-list.md` - короткий рабочий список задач AI.
- `tasks/task-details/<код>.md` - плоский файл детализации задачи.
- `tasks/task-draft.txt` - сырой или исторический буфер задач.
- `baseline/templates/task-details.md` - baseline-шаблон для нового task detail.

Важно: `baseline` хранит только правила и шаблоны. Реальные задачи, статусы и
детализация проекта не должны жить в `baseline/`.

## Что такое `task-details`

`task-details` - это готовая постановка для AI, а не свободные заметки.

Структура:

```text
ai-context/tasks/
  task-list.md
  task-draft.txt
  task-details/
    <код>.md
```

Каждая задача из `task-list.md` при детализации получает отдельный файл
`task-details/<код>.md`, созданный по шаблону
`ai-context/baseline/templates/task-details.md`.

Если задача крупная, ее декомпозиция, контекст, ограничения и связанные файлы
описываются прямо в `task-details/<код>.md`. Дополнительные поддиректории не
используются.

## Код задачи

По умолчанию используется локальный формат `01-001`, но допустимы и внешние
коды:

- локальный код репозитория, например `01-001`;
- Jira key, например `PROJ-123`;
- Bitrix task id, например `B24-12345`.

Если локальные стандарты репозитория не определены, использовать локальный
формат `01-001` как безопасный вариант по умолчанию.

## Карта статусов

- `🔴` - новая задача.
- `🟠` - задача детализирована.
- `🟡` - задача в работе.
- `🟣` - AI завершил работу и ждет ручной проверки пользователя.
- `🟢` - пользователь вручную проверил и подтвердил результат.

AI не должен переводить задачу в `🟢`.

## Команда `отформатируй задачи`

Если пользователь просит `отформатируй задачи`, нужно:

1. взять неотформатированные строки из `tasks/task-list.md` или из
   указанного сырого источника;
2. сохранить порядок задач;
3. добавить код всем задачам, у которых его нет;
4. добавить статус всем задачам, у которых его нет;
5. использовать статус `🔴` по умолчанию;
6. привести каждую строку к формату `🔴 [<код>] Короткое описание задачи`.

## Команда `детализируй <код>`

Если пользователь просит `детализируй <код>`, нужно:

1. найти задачу по коду в `tasks/task-list.md`;
2. создать файл `tasks/task-details/<код>.md`;
3. заполнить его по шаблону `baseline/templates/task-details.md`;
4. обновить статус задачи в `tasks/task-list.md` на `🟠`, если
   работа еще не началась.

При детализации:

- описать цель и ожидаемый результат;
- перечислить ограничения и связанные файлы;
- сослаться на применимые `rules/*.md`, если они есть;
- сослаться на применимые `mcp/*.md`, если задача зависит от repo-specific
  MCP или интеграций;
- если задача крупная, описать декомпозицию прямо внутри того же файла.

## Execution-правила

### Правило статусов задач

- После детализации переводить задачу из `🔴` в `🟠`.
- Когда задача берется в работу, допустимо ставить `🟡`.
- После завершения фактической работы AI должен ставить `🟣`, а не `🟢`.
- `🟢` ставит только пользователь после своей проверки.

Допустимые переходы:

- `🔴 -> 🟠`
- `🟠 -> 🟡`
- `🟡 -> 🟣`
- `🟠 -> 🟣`
- `🔴 -> 🟣` для маленькой задачи по прямому запросу пользователя
- `🟣 -> 🟢` только после ручной проверки пользователем

### Правило алерта о завершении задачи

После завершения задачи на macOS использовать только baseline-скрипт:

```bash
./ai-context/baseline/scripts/show-completion-alert.sh "Codex" "Работа завершена."
```

Текст в чате не считается заменой модальному окну.

### Стандарт резюме для коммита

Если в рамках задачи были file changes, в финальном ответе должна быть секция
`Резюме для коммита` и fenced `text` блок:

```text
<type>(<scope>): <краткий заголовок на русском> #ai

- <ключевое изменение 1>
- <ключевое изменение 2>
- <ключевое изменение 3>
```

Если изменений в файлах не было:

```text
коммит не требуется
```

### Правило changelog

После любых file changes нужно оставить запись в
`ai-context/changelog/`.

- одно завершенное изменение = один отдельный changelog-файл;
- рекомендуемый формат имени файла:
  `YYYY-MM-DD_HHMMSS_short-title.md`;
- после создания не дописывать в этот файл другие изменения;
- старые changelog-файлы не переписываются без прямого запроса пользователя;
- changelog-файл должен содержать время, заголовок, список затронутых файлов
  или областей и краткое описание фактически выполненных изменений.

Что запрещено:

- пропускать запись в changelog после реальных file changes;
- дописывать новые изменения в уже существующий changelog-файл;
- перезаписывать старые changelog-файлы вместо создания нового;
- писать абстрактные записи без перечисления фактических изменений.
