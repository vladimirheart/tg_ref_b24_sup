# Правило 012: AI-роллаут требует observability и offline-eval контура

## Статус

Действует.

## Правило

Если в проекте AI-агент принимает runtime-решения по клиентским обращениям, то перед расширением rollout должны существовать:

- decision trace API без необходимости читать сырые логи;
- offline evaluation dataset и регулярный прогон по нему;
- KPI-мониторинг, который показывает не только usage, но и quality/safety метрики;
- возможность отладить reclassify и retrieval отдельно от production диалога.

## Минимальный набор артефактов

- trace по конкретному dialog/ticket;
- offline eval set с репрезентативным числом кейсов;
- сохранение результатов регулярной оценки;
- KPI по ошибочным auto-reply, trust источника, stale evidence и review queue.

## Практический смысл

Без observability AI-контур начинает деградировать незаметно: thresholds меняются по ощущениям, rollout становится необратимым, а инциденты сложно локализовать. Обязательный trace и offline-eval делают качество управляемым инженерным процессом.
