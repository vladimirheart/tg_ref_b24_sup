## Что сделано

- Добавлен общий сервис `NotificationRoutingService`, который хранит и применяет гибкие правила маршрутизации bell-оповещений по модулям `tasks`, `passports` и `knowledge_base`.
- Добавлен API `GET/POST /api/settings/notification-routing` для чтения и сохранения глобальных правил оповещений через `NotificationRoutingSettingsController`.
- Подключено применение нового блока настроек в `SettingsTopLevelUpdateService`, чтобы конфиг маршрутизации жил в общем settings-контуре панели.
- Переведены уведомления задач, базы знаний, паспортов объектов и каталога ИТ-оборудования на новый роутинг-сервис вместо жёсткого `notifyAllOperators`.
- Для обращений через внешнюю форму расширена логика `AlertQueueService` и `OperatorNotificationWatcher`: добавлено отдельное событие `incomingClientMessage` с каналовой маршрутизацией bell-оповещений.
- В API канала расширена нормализация `questions_cfg.panelNotifications`: теперь сохраняются события `newPublicAppeal`, `incomingClientMessage`, `firstResponseOverdue` и единый routing-блок.
- В модалке настройки внешней формы канала добавлены поля для управления bell-оповещениями панели: события, отдел, режим получателей, явные сотрудники, исключения и режим доставки.

## Итог

- В проекте появилась единая основа для гибких bell-оповещений из нескольких доменов, а не только точечная логика для обращений.
- Для каналов операторских обращений теперь можно настраивать, кому именно приходят bell-оповещения по новым обращениям и новым сообщениям клиента: всем операторам, отделу или выбранным сотрудникам.
